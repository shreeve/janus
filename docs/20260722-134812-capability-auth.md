Cold Caddyfile capability. When enabled for a site, Janus is an
authentication wall in front of a tenant app that has none of its own:
every request must carry a valid session or it never reaches the app.
The canonical tenant is DuckDB's web query UI — a useful app with zero
login story — and the shape of the problem is always the same: small
user counts, one-off apps, an operator who wants "nobody but us" and
does not want to write auth into the app. auth is an **admission
wall, not an identity provider**: it answers "may this request pass,
and as whom," and nothing else.

| | |
| --- | --- |
| **Order** | **6** (after **ping**, **control**, **cache**, **hub**, **mdns**) |
| **Surface** | Cold config (Caddyfile) for the wall and its users; sessions and counters on hot `/1.0/auth` (observe and revoke only) |
| **Cascades** | **Yes** — global default → site override; explicit `off` beats inherited on; built-in default off (like **cache** and **hub**) |
| **Built-in default** | off (unset at every level → the site is exactly as open as today) |
| **Module** | Site handler check (data plane), between ping and hub interception; `auth.go`, `auth_config.go`, embedded `auth.html` (the `mdns.html` precedent); hot surface in `control_auth.go` per the `control_` prefix convention |
| **Directive** | Global `janus { auth { … } }` for the process default; site `janus { auth … }` to override |
| **Status** | **Contract — revision 2**; adversarial review folded in; the implementation lands against this revision |

**Naming.** The capability is `auth` — one short lowercase word beside
ping, control, cache, hub, and mdns, each named plainly for its
function. The reserved URL is ruled `/auth` (below), and the
capability's name agrees with its own front door: "the auth capability
serves `/auth`" needs no explanation, and that is the point. "guard"
was considered and rejected as the row's only metaphor; "sentry" was
rejected earlier for the sentry.io collision.

## Why it exists

- **Auth-less apps are a real and growing class.** Query UIs, admin
  panels, generated dashboards, one-off internal tools — apps whose
  value is the app and whose authentication story is "we never wrote
  one." Today the operator's choices are: expose it naked, wrap it in
  a VPN, or bolt a forward-auth sidecar onto the proxy. Janus already
  terminates TLS, admits the host, and owns the request path; one
  directive away from "and nobody unauthenticated gets in" is the
  natural place for the wall to live.
- **The edge is the correct trust boundary.** The app never sees a
  password, never stores a hash, never handles a session. It sees
  exactly one new fact: a `Remote-User` header naming who Janus
  admitted. Compromising the app yields no credentials, because the
  app never had any.
- **The logout story is the capability's reason to exist, stated
  plainly.** An auth-less app has no logout button and never will.
  With auth, the answer is one sentence: **go to `/auth` on any
  guarded site** — signed out, it is the login form; signed in, it is
  the status page with the sign-out button. That sentence is the
  entire user manual. Admins additionally revoke sessions over
  `/1.0/auth` (hot surface, below).
- **v3 already proved the flow.** The gate package
  (`rip-lang/packages/gate/index.rip`) shipped this exact product as
  Rip middleware: argon2id credentials, sliding-TTL sessions, signed
  double-submit CSRF, dummy-hash timing equalization, `safeReturnTo`,
  `Remote-User` forward-auth. This capability ports the proven flow
  into the edge, replacing the file-backed session store with the
  pooled in-memory state the Janus contract already mandates.

**Not an identity provider.** No roles, no groups, no OIDC, no OAuth,
no directory sync (Non-goals, below). One flat set of users, one
yes/no wall, one identity header. Operators who need an IdP have an
IdP; this capability is for the operator who needs a door.

## Scope

| Piece | Where | Role |
| --- | --- | --- |
| `janus { auth { … } }` | Global options `{ }` | Process-wide default: the wall's default posture and the default user set |
| `janus { auth … }` | Site block | Site override: `auth off` opts a site out; a site `auth { … }` block overrides per the cascade rules below |

**The perimeter hard rule.** Header-based identity is void if the app
is reachable around the edge. A guarded upstream must be reachable
**only through Janus** — in the house deployment that is automatic
(tenants register unix sockets or localhost-bound listeners), and the
capability doc states it as an operator obligation, not a hope: an
app that also listens on a routable address has no wall, whatever the
Caddyfile says. Janus cannot verify another process's bind addresses;
it can and does say this in plain words.

**One reserved URL: `/auth`, exact match only.** On a site whose
effective auth is on, Janus claims the exact path `/auth` and nothing
else — no prefix namespace, no underscore convention. `/auth/callback`,
`/authors`, `/auth.js` all proxy to the app untouched. The claim is
safe by the capability's own premise: guarded apps are by definition
auth-less, so `/auth` has no competing meaning on them — and it is the
URL a locked-out user would guess, short enough to speak aloud across
a room. On a site whose effective auth is off, `/auth` is an ordinary
app path and proxies like any other.

## Cascade

Site-scoped, the **cache**/**hub** pattern:

| Global | Site | Effective |
| --- | --- | --- |
| (unset) | (unset) | **off** |
| `auth { user … }` | (unset) | **on** (global users, global ttl) |
| `auth { user … }` | `auth off` | **off** for this site |
| (unset) | `auth { user … }` | **on** for this site |
| `auth { user alice …; ttl 8h }` | `auth { ttl 2h }` | on; global users; **2h** ttl for this site |
| `auth { user alice … }` | `auth { user bob … }` | on; **bob only** for this site (one level, no merging) |

Two resolution rules, distinct on purpose:

- **`ttl` cascades per key**, like every tuning knob in the house:
  site value → global value → built-in default (8h).
- **Users resolve at exactly one level — no merging.** A site block
  that declares at least one `user` line supplies the site's entire
  user set; a site block with no `user` lines inherits the global set
  whole. Merging user sets across levels is how an operator ends up
  unable to say which credentials open which door; one level always
  answers the question.

**The resolved set is a per-request authorization boundary, not a
login-time filter** (review ruling). On every guarded request, after
the session resolves, the wall verifies the session's user is in THIS
site's effective resolved set; a user outside it is treated as
unauthenticated (the 302/401 fork below), and the session itself stays
live for sites whose set does contain the user. A session minted on
site A never passes site B unless B's resolved set names that user —
site-level user replacement means what it says at every request, not
only at the login form.

**Hard error at handler provision: an enabled wall with zero
credentials.** If a site's effective auth is on and its resolved user
set is empty — no site `user` lines and no global ones — that
handler's provision fails loudly, naming both empty levels (site
identity is not computable at handler provision; the empty-level
naming is). An enabled wall with zero users is a lockout that looks
like a hang; rule 1 says reject, never tolerate. This is the
lifecycle split: per-site checks that are computable locally fire at
handler Provision; cross-site work — removed-user session revocation,
the reaper's ttl bound, the site table — runs at App Start (below).

**Global walls meet catch-all sites.** A global `auth { … }` is
inherited by EVERY janus site, including catch-alls — an on-demand
`https://` block and the mdns front-door carrier `http://*.local`
alike. The remedy is explicit: give catch-alls `auth off` unless
walling every unknown host is the intent. (The plain-HTTP carrier gets
the dead-wall treatment below either way.)

## Credentials

### The `g1:` format

Credentials are Janus-native compact blobs, **not** PHC strings:

```text
user alice g1:k5KZV0T/LsQSaqny8ZZa0hPn/FRGIDueMSt91JUSJzVp2TcbSNt3Hdo0RuBCMWV1
```

- The KDF is **argon2id** with parameters fixed as module constants
  under the `g1` version tag: **m = 64 MiB, t = 2, p = 1, 16-byte
  salt, 32-byte key**.
- The stored form is `g1:` + one **unpadded standard-alphabet base64**
  blob (`base64.RawStdEncoding`) encoding `salt‖digest` — exactly 48
  bytes decoded (16 + 32), which is exactly **64 encoded characters**,
  nothing else. Padding (`=`) and the URL alphabet (`-`, `_`) are
  rejected at parse by name — one encoding, no variants.
- **Why not PHC.** auth mints and verifies its own credentials; there
  is no interchange with other systems, so PHC's self-describing
  parameter envelope is unneeded noise. Fixing the parameters as
  constants deletes two attack classes outright: the
  hostile-parameter DoS (a "credential" declaring m=4GiB that the
  verifier obediently honors) and the PHC-parser attack surface
  itself. There is nothing to parse but base64 and a length.
- **Future cost changes mint `g2`** with its own fixed constants,
  verified side by side with `g1`; existing credentials keep working
  and re-mint at the operator's pace. The tag is a format version,
  not a name.
- **Malformed blobs reject at parse/provision, never at first
  login:** bad base64, padding or URL-alphabet characters, a decoded
  length other than 48 bytes, an unknown version tag — each is a
  precise Caddyfile error at adapt time. A wall that discovers its
  credentials are broken when the first user tries the door is
  drift-tolerance.

### Minting: the `janus-auth-hash` command

The janus-built `caddy` binary gains a **`janus-auth-hash`** command
(via `caddycmd.RegisterCommand`; the name is namespaced — a generic
top-level `auth` command would risk colliding with upstream or other
modules' command names, per the review's Q10 ruling):

```bash
./bin/caddy janus-auth-hash
# Password: (no echo)  Confirm: (no echo)  → prints one line:
# g1:k5KZV0T/LsQSaqny8ZZa0hPn/FRGIDueMSt91JUSJzVp2TcbSNt3Hdo0RuBCMWV1
```

The password never rides argv (shell history is not a credential
store). On a real terminal it is prompted without echo and confirmed;
when stdin is **not** a terminal, exactly one line is read from stdin
— so `printf 'pw\n' | ./bin/caddy janus-auth-hash` works in scripts
and test fixtures. Output is the one line the operator pastes after
`user <name>`. The command runs the same constants the verifier runs
— there is exactly one definition of `g1` in the codebase.

## The `/auth` endpoint

One URL, one embedded page (`auth.html`, `go:embed`, zero external
resources — the `mdns.html` precedent), two visual states (login form
/ signed-in status), and a four-row state machine.

**The POST dispatch predicate** (review ruling, replacing prose): a
POST to `/auth` is a **sign-out iff neither a `user` nor a `password`
field is present** after parsing exactly one
`application/x-www-form-urlencoded` body; otherwise it is a sign-in
attempt. Empty-but-present values are sign-in attempts that fail
verification (401 re-prompt), never sign-outs — a stale login tab left
open by a user who signed in elsewhere still reads as a login attempt,
never as an accidental sign-out. Before the predicate runs:

- A non-urlencoded `Content-Type` — multipart, JSON, or none — is
  **415**; there is exactly one body shape.
- The 64 KB body cap is enforced with `http.MaxBytesReader` **before**
  `ParseForm`; an oversized or unparseable body is **400**.
- A duplicated `user`, `password`, or `csrf` field is **400** — a
  request that says two things is answered with neither.

| Request | Session | Behavior |
| --- | --- | --- |
| `GET /auth` (or `HEAD`) | none / invalid | **Login form**: username, password, CSRF token (hidden field + short-lived CSRF cookie), optional `to` carried through from the query string (sanitized). The page brands itself with the site hostname — "Sign in to duckdb.ripdev.io" — which is why no `realm` knob exists. `HEAD` never sets cookies |
| `POST /auth`, sign-in per the predicate | any | **Sign-in attempt**: CSRF validated first (mismatch/missing → 403); cheap caps next; throttle next (below), before any argon2 work; then verification behind the semaphore. Success mints a **fresh** session token — a client-proposed token is never accepted, so session fixation is dead by construction — sets the session cookie, clears the CSRF cookie, and answers **303** to the sanitized `to` (default `/`). Failure re-renders the login form with a generic "invalid credentials" (401); which half was wrong is never disclosed. The re-rendered form reuses the submitted CSRF pair — the cookie is untouched and still within its 600 s window, so the retry stays coherent |
| `GET /auth` (or `HEAD`) | valid | **Status page**: "Signed in as alice … since 09:14" plus the sign-out button (a POST form carrying a CSRF token). The page **re-mints and re-plants a fresh `__Host-janus_csrf` cookie** with its matching form field — login cleared the login form's CSRF cookie, and a sign-out form without a live pair would 403. This page **is** the logout UI for every guarded app |
| `POST /auth`, sign-out per the predicate | any | **Sign-out**: CSRF validated (403 without); the session named by the cookie is revoked server-side, both cookies are cleared, and the response is a 303 redirect to `/auth` — which now renders the login form. Signing out an already-dead session is a success, not an error |

Any other method on `/auth` → **405** with `Allow` — and the method
table takes precedence over the wall's 401: `/auth` is claimed, and a
claimed URL answers its own grammar. The username is capped at 256
bytes and the password at 1024 before any hashing (v3's
cheap-caps-before-argon2 rule). Every response this endpoint emits
carries `Cache-Control: no-store`.

**One stated limitation (v3 parity):** a non-`/auth` POST whose
session expired mid-composition gets the bare 401 and the submission
is lost — the wall does not buffer request bodies through a re-login.

**CSRF: signed double-submit, v3's scheme.** The token is
`nonce.HMAC(key, nonce)` with the per-boot key (below;
domain-separated from the session-key HMAC). GET plants it in both a
hidden form field and a short-lived (600 s, Max-Age'd)
`__Host-janus_csrf` cookie; POST requires cookie == field and a
valid HMAC, compared in constant time. The HMAC stops cookie forgery;
the cookie/field match with SameSite=Lax keeps cross-site POSTs out;
no server-side state is needed, which suits a pre-auth form. A
process restart rotates the key, so login forms rendered before a
restart fail CSRF after it — the user reloads the form and signs in;
consistent with restart-wipes-sessions (below).

## Unauthenticated requests

Every request to a guarded site that carries no valid session — or a
valid session whose user is outside this site's resolved set — any
path, any method, except exact `/auth` and the exempt liveness paths
`/ping` and `/ping/` (matching the handler's own ping table; ordering,
below):

| Request shape | Response |
| --- | --- |
| Browser-shaped: method `GET`/`HEAD` **and** `Accept` contains `text/html` (a substring test) **and** not a WebSocket upgrade | **302** to `/auth?to=<original path+query>` — the human lands on the login form and returns to where they were going |
| Everything else — API calls, fetches, curl, WebSocket upgrades | **401** with a plain body and **no `WWW-Authenticate` header** (the wall speaks cookies, not Bearer — unlike the control plane's `bearerAuth`). A WS upgrade never gets a redirect it cannot follow, whatever its `Accept` says |

Every pre-auth response — the 302, the 401, and every `/auth` page —
carries **`Cache-Control: no-store`**, so the redirect-to-login is
never storable and the "did the cache trap someone on the login
bounce" question never arises. (The micro-cache would refuse most of
these anyway — non-200, and `/auth` responses set cookies — but the
wall does not rely on another capability's table; it states its own.)

**`to` sanitization (`safeTo`) is ported exactly from v3's
`safeReturnTo`** and pinned with a table-driven test: the value must
be a same-origin path — a string under 2048 bytes starting with a
single `/`, rejecting `//` and `/\` prefixes (protocol-relative and
backslash-scheme escapes), and rejecting any control character,
whitespace, or backslash anywhere. Anything that fails any check
collapses to `/`. There is no open-redirect budget: the redirect
target is always this site.

## Sessions

**Store: pooled, in-memory, keyed by HMAC of the token.**

- The session store lives in the pooled process state
  (`caddy.UsagePool`, the `janusState` holder in `state.go`) beside
  the registry, data plane, hubs, and the mdns advertiser: **a config
  reload never logs anyone out; a process restart wipes every session
  and everyone signs in again.** v3 said it first and this contract
  repeats it in v3's words: a reboot forcing re-login is a feature,
  not a loss — and it is exactly what the memory-only contract
  already promises for the registry.
- The session token is 128 bits from `crypto/rand`, base64url, minted
  fresh at every successful login (v3's `newToken`). The store maps
  **HMAC(per-boot key, token) → {user, issuedAt, lastSeen}** — the
  raw token exists only in the client's cookie. Lookup hashes the
  presented cookie value and does a map get: timing tells an attacker
  nothing (the hash is constant-work, the map is keyed by an
  attacker-unpredictable digest), and a memory disclosure of the
  store yields no replayable tokens.
- **The per-boot key** is 32 bytes from `crypto/rand`, minted once
  when the pooled holder is constructed, used for session-key HMAC
  and CSRF signing. There is no `secret` knob: nothing about the key
  needs to survive a restart (sessions don't), so nothing about it
  belongs in config.
- **Sliding idle TTL, server-side only.** `lastSeen` is bumped on
  every authenticated request; a session whose idle age exceeds the
  effective `ttl` (default 8h) is dead. The cookie is session-scoped
  (no Max-Age) precisely so the server's clock is the sole expiry
  authority — v3's rule, kept: a stolen-but-idle cookie expires here
  even though the browser still holds it.
- **Expiry enforcement is lazy plus a slow reaper:** a lookup that
  finds an expired entry deletes it; a background tick (registry-
  sweeper cadence class, minutes not milliseconds) sweeps idle
  corpses so an abandoned store does not grow forever. Neither path
  touches a request that isn't already paying for a lookup.
- **Reload revokes a removed user's sessions — at App Start.** A
  session is not a credential — but a user deleted from the Caddyfile
  must not stay signed in until their TTL runs out. At **App Start**
  (past the reload's point of no return, the mdns introspection
  precedent — never per-handler Provision, so an **aborted reload
  never logs users out**), the new config's resolved user sets are
  compared against live sessions; sessions whose user no longer
  appears in any enabled site's resolved set are revoked, counted,
  and logged. The residue is the mdns no-rollback shape: a reload
  that passes janus Start and aborts elsewhere has already run this
  revocation. With the per-request set check (Cascade, above) this
  revocation is promptness, not the security boundary. Stated here
  and pinned in tests.

**Cookie: `__Host-janus`, unconditionally.** `Secure`,
`HttpOnly`, `SameSite=Lax`, `Path=/`, session-scoped, no `Domain` —
the `__Host-` prefix makes the browser enforce exactly that shape.
There is no prod/dev split and no `cookie` knob: guarded sites are
always HTTPS behind Janus (the data plane's sites are TLS-terminated;
the mdns plain-HTTP front door is out of scope below), so the Secure
prefix always holds. One fixed name means one line of debugging
("look for `__Host-janus`") forever.

**The wall's cookies are stripped before the request proceeds**
(review ruling, Q8): `__Host-janus` and `__Host-janus_csrf`
are removed from the proxied upstream request AND from the hub's
handshake-header snapshot before it freezes — a compromised or
log-happy upstream never holds a live bearer token for its own wall.
Every other cookie rides through untouched. The critical corollary:
the micro-cache's authenticated-traffic bypass is an **explicit
internal signal decided at the auth step** (a request-context value
set on the fall-through), never an inference from Cookie-header
residue — naive stripping would have made per-identity responses
cacheable and leaked them to anonymous users (Cache interplay,
below).

## Login hardening

The v3 flow plus the good ideas from the abandoned v4 stub, in
checking order:

1. **CSRF first** (above): no valid double-submit pair → 403 before
   anything else runs.
2. **Throttle before the burn.** A fixed-window counter per
   **(client key, username)**: **5 failed attempts per 15 minutes** →
   **429** with `Retry-After`, checked **before** any argon2 work, so
   a brute-force loop costs Janus a map lookup, not 64 MiB of KDF
   memory per guess. The client key derives from the connection's
   `RemoteAddr` — Janus is the TLS-terminating edge, so that is the
   true peer — and never `X-Forwarded-For`, which is
   attacker-supplied ink. **IPv6 clients key on the /64 prefix**
   (rotating within a delegated subnet buys nothing); IPv4 keys on
   the address. A successful login clears the pair's counter; windows
   expire and their entries are swept with them. **The throttle map
   is bounded at 65,536 entries**: at the cap, expired windows are
   swept first and, failing that, an arbitrary entry is evicted — a
   distributed (client, username) spray cannot grow the map without
   bound.
3. **Dummy-hash timing equalization.** An unknown username costs the
   same argon2 verification as a wrong password — a dummy credential
   minted lazily on first miss (v3's `verifyMatches`), so response
   timing never enumerates the user list.
4. **Constant-time comparisons everywhere a secret is compared** —
   the house pattern is `bearerAuth` in `control_api.go`
   (`subtle.ConstantTimeCompare`); the CSRF HMAC check and the
   argon2 digest check use the same discipline.
5. **A small semaphore caps concurrent argon2 verifications** (fixed
   at 2 in v1) so a login flood — even one that beats the throttle by
   spraying (client, username) pairs — cannot stack 64 MiB
   allocations and starve the data plane. **The queue behind the
   semaphore is bounded at 16**: an attempt arriving past that depth
   fast-fails 429 instead of queueing — a distinct-pair spray can
   never hold unbounded goroutines waiting for KDF slots.

Counters for every one of these outcomes land on `/1.0/auth` (hot
surface, below) — a wall whose rejections are invisible is a wall
nobody can operate.

**Memory floor, stated plainly:** argon2 verification peaks at
2 × 64 MiB (the semaphore bound × the g1 memory parameter) beside the
data plane's own working set. A guarded deployment needs headroom
comfortably above 128 MiB; on smaller boxes the wall still works —
logins simply serialize behind the semaphore.

## Identity transport

- **Strip, then inject.** On every request to a guarded site, auth
  unconditionally deletes any client-supplied `Remote-User` header
  (a spoofed identity from the outside must die at the edge whether
  or not a session is present), and for an authenticated session
  injects **`Remote-User: <name>`** on the proxied request. The
  username character set (Legal lines, below) is header-safe by
  construction, so injection is inert.
- **The hub bridge snapshot carries it too — and never the cookie.**
  The hub's handshake header snapshot (`hubHeaderSnapshot` in
  `hub_bridge.go`) is filtered and frozen at open and replayed on
  every bridge POST; on a guarded site the same strip-and-inject runs
  before the snapshot freezes, so the tenant sees `Remote-User` on
  `open`/`text`/`close` bridges exactly as it does on proxied
  requests — and never `__Host-janus` — symmetric on purpose,
  because the tenant cannot resolve the session cookie itself
  (sessions live in Janus memory) and the bridge is the only identity
  channel a WebSocket has.
- **The perimeter rule, restated where it bites:** `Remote-User` is
  trustworthy exactly as far as the only road to the app runs through
  Janus. Scope section, operator obligation, plain words.

## Handler ordering (pinned)

The site handler's `ServeHTTP` (`handler.go`) runs, in order: the
mdns shared front-door decider (~line 98), ping (~line 106), hub
upgrade interception (~line 116), then cache/data plane (~line 119).
**The auth check slots between ping and hub interception:**

1. **mdns front-door decider** — the plain-HTTP status page is
   process-wide and outside this capability (Non-goals). On the
   shared front-door port, auth is **inert by construction**: a
   front-door Host is served by the decider and a not-mine Host
   returns via `next.ServeHTTP` into Caddy's auto-HTTPS redirects —
   neither ever reaches the auth step.
2. **ping** — **`/ping` (and `/ping/`) answers unauthenticated on
   guarded sites** (ruled): it is the liveness probe, it discloses
   one static word, and a monitor that needs a session is a monitor
   that pages you about auth instead of the app. Ping off makes
   `/ping` an ordinary walled path.
3. **auth** — first, the **plain-HTTP dead wall** (below): a guarded
   request with no TLS answers **421**. Then exact `/auth` serves the
   endpoint above; any other request without a valid, site-authorized
   session gets the 302/401 treatment; a valid session bumps
   `lastSeen`, strips the wall's cookies, strips/injects
   `Remote-User`, sets the cache-bypass signal, and falls through.
4. **hub interception** — a WS upgrade on the hub path therefore
   flows through the same auth check first: no session → **401**
   (never 302 — an upgrade cannot follow a redirect). The tenant
   obligation: a WS client whose session died mid-life sees its
   reconnect answered 401 and must send the human to `/auth` itself —
   the wall cannot redirect an upgrade.
5. **cache, then the data plane** — unchanged.

**The plain-HTTP dead wall (the review's DB-3, superseding revision
1's open question 11).** A static provision/Start check cannot
distinguish the legal plain-HTTP front-door carrier
(`http://*.local { janus }`) from a genuine plain-HTTP data-plane
site, and would reject `Caddyfile.example`. So the check is
per-request and isolated in one small function (swappable on an owner
re-ruling): at the auth step, a request on a guarded site with
`r.TLS == nil` answers **421 Misdirected Request** with the plain body
`auth requires HTTPS; the __Host- session cookie cannot be set over
plain HTTP`, plus a **once-per-site ERROR log** naming the host and
both remedies (`auth off` or HTTPS). The correction to revision 1's
OQ11 prose: on the shared front-door port, not-mine Hosts return via
`next.ServeHTTP` into the auto-HTTPS redirects and never reach auth —
the dead-wall victims are plain-HTTP janus **data-plane** sites (mdns
off, dedicated mode, or a non-shared port) whose effective auth is
on.

## Cache interplay (verified against the shipped cache)

One new row of machinery (the strip ruling's corollary), the rest the
existing contract observed from the wall's side, with the seams cited:

- **Authenticated traffic bypasses on the wall's EXPLICIT signal.**
  Because the wall strips its own cookies before the request proceeds,
  the old inference — "a Cookie header means bypass" — would no longer
  fire for a request whose only cookie was the session, and a
  per-identity response could be stored and leaked to anonymous
  users. So the auth step sets a request-context signal on every
  authenticated fall-through, and `cacheBypassRequest` (`cache.go`)
  checks it first: **authenticated traffic always bypasses**, decided
  at the auth step, never inferred from header residue. Other
  cookies still ride through to the upstream and still trip the
  existing Cookie bypass row on their own.
- **`Set-Cookie` responses are never stored** — `evaluateHeaders`
  (`cache_serve.go` ~line 505). The login 303 and sign-out 303 set
  cookies and could never enter the store even if they were 200s.
- **Pre-auth responses are `no-store` by auth's own discipline**
  (above) — the wall does not depend on the previous rows for its
  redirects; it vetoes storage explicitly.

The one direction that matters for operators: enabling auth on a
cached site effectively disables the cache for signed-in users
(cookie bypass) — which is correct, because every response behind the
wall is per-identity by assumption. The capability doc says so; no
knob changes it.

## Cold config

### Syntax

Normal Caddyfile grammar — exactly three knobs:

```caddyfile
{
	janus {
		control local
		auth {                              # global default: wall on
			user alice g1:k5KZV0T…RuBCMWV1  # repeatable; 64 unpadded base64 chars
			user bob   g1:ZO7UHjS…5EO2EW6a
			ttl 8h                          # sliding idle timeout (default)
		}
	}
}

(ripdev_tls) {
	tls certs/ripdev.io.crt certs/ripdev.io.key
}

duckdb.ripdev.io {                          # inherits the wall + global users
	import ripdev_tls
	janus
}

status.ripdev.io {
	import ripdev_tls
	janus {
		auth off                            # explicit off beats inherited on
	}
}

notes.ripdev.io {
	import ripdev_tls
	janus {
		auth {
			user carol g1:vubzMm3…jGXXYa    # site users REPLACE global users
			ttl 30m                         # site ttl overrides global
		}
	}
}
```

### Legal lines

```text
auth
auth on
auth off                        # explicit opt-out; legal at both levels
auth {
	user <name> g1:<base64>     # repeatable; name: [a-z0-9._-]{1,64}, lowercased
	ttl <duration>              # > 0; sliding idle timeout
}
auth on { … }                   # same as auth { … }
```

A global `auth off` follows the cache/hub precedent for a global off
line: it parses, sets the process default to off, and means exactly
what an absent directive means unless a site says otherwise.

Usernames are lowercased on parse (login lowercases the submitted
name the same way — v3's rule), limited to `a-z 0-9 . _ -` and 64
bytes: header-safe for `Remote-User` by construction, no encoding
questions anywhere downstream.

### Defaults

| Knob | Default |
| --- | --- |
| capability | off (unset at every level) |
| `ttl` | 8h (v3's 28800 s) |
| Cookie | `__Host-janus` — fixed, no knob |
| CSRF cookie | `__Host-janus_csrf`, 600 s — fixed |
| KDF | argon2id, m = 64 MiB, t = 2, p = 1, salt 16 B, key 32 B — the `g1` constants, not configurable |
| Login throttle | 5 failed attempts / 15 min per (client key, username) → 429; IPv6 keys on the /64 prefix (fixed in v1) |
| Throttle map bound | 65,536 entries (expired swept first, then arbitrary eviction; fixed in v1) |
| Argon2 concurrency / queue depth | 2 / 16 — past the queue, fast 429 (fixed in v1) |
| POST body cap / username cap / password cap | 64 KB / 256 B / 1024 B (fixed in v1) |
| `to` length cap | 2048 B (fixed in v1, v3's value) |

### Hard errors (reject at parse)

- Unknown argument (anything but `on` / `off`)
- `auth off { … }` — a block on an off switch is a contradiction
  (the cache rule, verbatim shape)
- Duplicate `auth` directive in the same block; duplicate `ttl`
- Unknown subdirective inside the block; nested block under any
  subdirective
- `user` without exactly two arguments
- A duplicate username within one block
- An illegal username: empty, > 64 bytes, or any character outside
  `a-z 0-9 . _ -` after lowercasing
- A malformed credential: missing `g1:` prefix, an unknown version
  tag, invalid base64, padding (`=`) or URL-alphabet (`-`, `_`)
  characters, or a decoded length ≠ 48 bytes — each named precisely
  (`user alice: credential is not a g1 blob (decoded 33 bytes, want
  48)`)
- `ttl` not a valid positive duration

### Hard errors (refuse at handler provision)

- **Any site whose effective auth is on with an empty resolved user
  set** — the lockout rule (Cascade, above); the error names both
  empty levels. This is the per-site, locally computable half of the
  lifecycle split.

### At App Start (past the point of no return)

- Removed-user session revocation and the reaper's ttl bound
  (Sessions, above) — cross-site work, deliberately after the reload
  can no longer abort into a logged-out fleet.
- **No plain-HTTP provision or Start error exists** (review ruling,
  DB-3): the dead wall is the per-request 421 (Handler ordering,
  above), because a static check cannot tell the legal plain-HTTP
  front-door carrier from a genuine plain-HTTP data-plane site.

## Behavior contract

### The wall

| Row | Contract |
| --- | --- |
| Off | Effective auth off → the site behaves byte-for-byte as today; `/auth` is an ordinary app path |
| Plain HTTP | Guarded site, `r.TLS == nil` → 421 + plain body + once-per-site ERROR (the dead wall) |
| No session, browser-shaped | GET/HEAD (not an upgrade) with `Accept` containing `text/html` → 302 to `/auth?to=<path+query>`, `no-store` |
| No session, everything else | 401, `no-store`, no `WWW-Authenticate` — including WS upgrades on the hub path |
| Valid session, user in this site's set | `lastSeen` bumps (sliding); client `Remote-User` stripped; `Remote-User: <user>` injected; the wall's cookies stripped; the cache-bypass signal set; request proceeds to hub/cache/data plane |
| Valid session, user NOT in this site's set | Treated as unauthenticated (302/401 fork); the session itself stays live for its own sites |
| Any session state | Client-supplied `Remote-User` is stripped unconditionally, session or not |
| `/ping`, `/ping/` | Answers unauthenticated when the site's effective ping is on (liveness is not a secret); an ordinary walled path when ping is off |
| `/auth` | Exact match only; the state machine above; every response `no-store`; other methods 405 + `Allow` (the method table beats the wall's 401) |

### Sessions

| Row | Contract |
| --- | --- |
| Mint | Only at successful login; 128-bit `crypto/rand` token; never client-proposed (fixation defense) |
| Store | Pooled state; keyed by HMAC(per-boot key, token); value {user, issuedAt, lastSeen} |
| Expiry | Sliding idle `ttl`, server-enforced; lazy delete on lookup + slow reaper tick |
| Config reload | Sessions survive (pooled state); sessions of users absent from every enabled site's resolved set are revoked at App Start, counted and logged (an aborted reload never revokes) |
| Process restart | Every session gone; every user signs in again; the per-boot key rotates, so pre-restart CSRF tokens also die. A feature, not a loss |
| Revocation | Sign-out (self), `/1.0/auth` DELETE (admin), reload user-removal (automatic) |

### Security

| Row | Contract |
| --- | --- |
| Passwords | argon2id under the `g1` constants; verified behind the throttle and the semaphore; dummy-hash equalization for unknown users; constant-time digest compare |
| CSRF | Signed double-submit on both POST arms (login and sign-out); per-boot key; constant-time compare |
| Session fixation | Impossible by construction: tokens are only ever minted server-side at login |
| Open redirect | Impossible by construction: `safeTo` admits same-origin paths only, collapses everything else to `/` |
| Enumeration | Login failure is one generic message; unknown-user timing equalized; the store is keyed by HMAC so lookups leak nothing |
| Cookie theft | Bounded by the sliding `ttl`; an idle stolen cookie dies server-side; admins revoke live ones over `/1.0/auth`; the upstream never sees the cookie at all (stripped) |
| Header spoofing | Client `Remote-User` dies at the edge on every request to a guarded site |
| Site authorization | Per request: the session's user must be in THIS site's resolved set, or the request is unauthenticated |
| Perimeter | Stated operator obligation: guarded upstreams reachable only through Janus (bind localhost / unix sockets) |
| DoS | Throttle before the KDF burn (IPv6 /64 keying; bounded map); argon2 semaphore (2) with a bounded queue (16, then fast 429); body/field caps before hashing; 429 + `Retry-After` |

## Hot surface (`/1.0/auth` — observe and revoke, never configure)

In `control_auth.go` per the `control_` prefix convention, on every
control listener with the existing Bearer behavior. **Hot JSON never
adds users, never changes `ttl`, never toggles the wall** — the cold
admission gate is cold (rule 2). Registry-delivered credentials are
explicitly rejected as a non-goal below.

- **`GET /1.0/auth`** — always answers; `{"enabled": false}` when no
  site's effective auth is on (the `/1.0/mdns` precedent):

  ```json
  {
    "enabled": true,
    "sites": ["duckdb.ripdev.io", "notes.ripdev.io"],
    "sessions": 3,
    "logins": 17, "login_failures": 4, "throttled": 2,
    "signouts": 5, "revoked": 1, "reload_revoked": 0,
    "expired": 8
  }
  ```

  Counters are monotonic, not mutually atomic (the `/1.0/cache`
  discipline). Usernames of configured users are deliberately absent
  here — the wall's user list is cold config, not a hot inventory.

- **`GET /1.0/auth/sessions`** — the live list:

  ```json
  {
    "sessions": [
      { "id": "3f81c29aa04d", "user": "alice",
        "age_ms": 5400000, "idle_ms": 12000 }
    ]
  }
  ```

  `id` is a 12-hex-char prefix of the **HMACed store key** — Janus
  does not retain raw tokens, so no listing can leak one; the prefix
  identifies a row to a human and to the DELETE below, and disclosing
  it grants nothing (the store key is not the credential).

- **`DELETE /1.0/auth/sessions/{id}`** — revoke one session. The `id`
  prefix must resolve a **unique** full store key: no match → 404, an
  ambiguous prefix → 409, unique → 204.
- **`DELETE /1.0/auth/sessions`** — wipe all sessions ("kick
  everyone"; v3's `rm <dir>/*`, one HTTP call); answers the count.

`GET /1.0` gains an `"auth"` boolean beside `"ping"` and `"mdns"` —
true when any site's effective auth is on (the mdns
deliberate-asymmetry precedent: presence in the live config, not a
global-default echo).

## Non-goals (v1)

Each a decision, not an omission:

- **No identity-provider features** — no roles, groups, OIDC, OAuth,
  SAML, directories, or per-user anything beyond a name and a hash.
  Operators who need an IdP have one; this is a door.
- **No registry-delivered credentials** — a tenant cannot POST users
  into the wall. Users are cold config only; the hot plane observes
  and revokes (rule 2, restated as a refusal).
- **No guarding the mdns front door** — the mdns contract rules its
  status page interface-public-or-disabled for v1; an authenticated
  dashboard is named future work there, and this capability does not
  smuggle it in.
- **No HTML rewriting** — no injected logout bars or identity chrome
  in app pages. Rewriting tenant HTML is invasive and breaks
  CSP/streaming; the `/auth` status page **is** the logout UI.
- **No guarding `/1.0` control listeners** — control has its own
  Bearer story (`token:` per listener); mixing the two walls would
  blur which credential opens which plane.
- **No persistent sessions** — nothing on disk, nothing across
  restarts, per the memory-only contract and v3's design alike.
- **No `realm`, `cookie`, or `secret` knobs** — the login page brands
  itself with the hostname; the cookie name is fixed; the key is
  per-boot random. Three knobs is the grammar.
- **No forward-auth mode** — v3's `/check` endpoint existed to serve
  an external proxy; Janus **is** the proxy, so the mode has no
  caller here.

**Future work** (explicitly deferred, named): guarding the mdns front
door once its contract asks for it; a `g2` cost bump when hardware
argues for one.

## Acceptance sketch (`test.sh` auth group)

Appended after `mdns` (groups run in capability landing order). The
instrument: a fixture upstream that records received headers (the
tenant-side truth: Remote-User and Cookie on every request and on hub
bridge POSTs) plus `/1.0/auth` (the Janus-side truth). Cookies are
carried explicitly per request; the minter is exercised end-to-end
(`printf 'pw\n' | caddy janus-auth-hash` → the blob opens a wall in a
sockets-apart fixture caddy).

| Case | Assert |
| --- | --- |
| **Wall blocks** | Guarded site, no cookie: `GET /` with `Accept: text/html` → 302 to `/auth?to=%2F` + `no-store`; `GET /api` without it → 401; upstream request counter unmoved |
| **Login round trip** | GET `/auth` (form + CSRF cookie) → POST valid credentials → 303 to `/` + `__Host-janus`; follow-up `GET /` reaches the upstream, which sees `Remote-User: alice` and **never the session cookie** (stripped) |
| **`to` carried** | 302's `to` survives login: POST from `/auth?to=%2Freports%3Fq%3D1` lands 303 on `/reports?q=1` |
| **`to` sanitized** | `to=//evil.example` and `/\evil` and `https://evil` all land on `/` |
| **Status page** | Signed in, `GET /auth` → 200 with "Signed in as alice" and the sign-out form, with a **re-minted CSRF pair** (cookie == field) so the sign-out works after login cleared the login form's cookie |
| **Sign-out** | POST `/auth` (no user/password fields, valid CSRF) → 303 `/auth`; cookie cleared; next `GET /` → 302 again; `/1.0/auth` signouts moved |
| **CSRF enforced** | POST login with a missing/mismatched CSRF pair → 403; nothing minted |
| **Spoofed header dies** | Request with `Remote-User: root` and no session → 401; with a valid session → upstream sees `Remote-User: alice`, never `root` |
| **Fixation dead** | A pre-set `__Host-janus` cookie value is never honored and never re-minted verbatim after login |
| **Wrong credentials** | Bad password and unknown user → identical 401 bodies; `login_failures` moves |
| **Throttle** | 6th failed attempt inside the window → 429 + `Retry-After`; `throttled` moves; a valid login from an unthrottled pair still works |
| **`/ping` open** | Guarded site with ping on: `GET /ping` → 200 `pong`, no session |
| **`/auth` exact only** | `GET /authors` and `/auth/callback` on a guarded site → proxied to the upstream (which records them), not the login form |
| **WS 401** | Upgrade to the hub path without a session → 401, no connection; with a session → connected, and the tenant's bridge `open` snapshot carries `Remote-User` and never `__Host-janus` |
| **Cache bypass holds** | Guarded + cached site: signed-in requests never HIT (the explicit auth signal — the strip removed the Cookie header the old rule keyed on); the 302-to-login is `no-store` |
| **Cascade + per-request authorization** | Site user-set replacement observed twice: global alice's CREDENTIALS fail on the carol-only site, and alice's LIVE SESSION from the inheriting site is redirected to login there; carol succeeds |
| **Minter + dead wall** | `printf 'pw\n' \| caddy janus-auth-hash` emits one unpadded 64-char `g1:` line; a fixture caddy configured with it completes the login round trip; its plain-HTTP guarded site answers **421** with the HTTPS-remedy body and ERROR-logs once |
| **Reload keeps sessions** | Sign in; reload an identical config; the session still works (no re-login) |
| **Reload revokes removed user** | Sign in as bob; reload with bob's `user` line deleted; bob's next request → 302; `reload_revoked` moved |
| **Hot revoke** | `GET /1.0/auth/sessions` lists the session; `DELETE …/sessions/{id}` → that cookie is dead; `DELETE …/sessions` after two logins → both dead |
| **Restart wipes** | Stop and start the process; the old cookie → 302 (fresh registry, fresh sessions) |
| **Zero-users lockout** | A config with `auth on` and no `user` line anywhere fails `caddy validate` with the empty-set error |
| **Parse rejections** | Each parse hard-error line above fails `caddy adapt` with a precise error (bad blobs, bad names, duplicate users, `auth off { … }`, bad `ttl`) |

### Go-test surface

`go test ./...` units, one per contract row:

- **Parse table** — every legal line accepted; every hard-error row
  rejected with its message; username lowercasing; duplicate
  detection; `auth off { … }` contradiction.
- **`g1` codec** — mint/verify round trip; the truncated-blob,
  bad-base64, wrong-length, and unknown-tag rejections; the digest
  compare is `subtle.ConstantTimeCompare`.
- **`safeTo` table** — ported from v3's `safeReturnTo` and extended: empty,
  non-string-shaped, no leading slash, `//`, `/\`, control chars,
  whitespace, backslash, length cap, plus the legal
  path-with-query pass-throughs.
- **State-machine dispatch** — the predicate's torture shapes pinned:
  user-only, password-only, empty-string values (sign-in attempts,
  never sign-outs), duplicated fields → 400, multipart / JSON /
  missing Content-Type → 415, oversized body → 400 before parsing,
  the stale-login-tab case (signed-in + POST with user/password =
  login attempt, session untouched); 405 table with `Allow`;
  `no-store` on every arm; HEAD sets no cookies.
- **Per-request site authorization** — a session minted on the
  alice-site is unauthenticated on the carol-site and stays live for
  its own site.
- **Cascade resolution** — the cascade table above, including
  one-level user replacement (never merged) and per-key `ttl`.
- **Zero-users provision error** — enabled site + empty resolved set
  at each level combination.
- **Session store** — mint/lookup/revoke; sliding bump; idle expiry;
  lazy delete; reaper sweep; HMAC keying (raw token never stored);
  Start user-removal revocation; wipe-all; id-prefix resolution
  (unique / 404 / ambiguous).
- **Throttle** — window arithmetic; per-(client, user) isolation;
  IPv6 /64 keying; the map bound; check ordering pinned (a throttled
  attempt performs zero argon2 work — instrumented KDF counter);
  success clears; entries expire.
- **Timing equalization** — unknown user and wrong password take the
  same code path (instrumented: exactly one KDF verification each).
- **Semaphore + queue** — at most 2 concurrent verifications under a
  flood; admissions past the queue depth fast-fail (`-race`).
- **Header + cookie transport** — strip-unconditionally;
  inject-on-session; the wall's cookies stripped while other cookies
  ride; the hub snapshot carries the injected value and never a
  client forgery or the session cookie.
- **Cache signal** — the explicit context signal bypasses with no
  Cookie header present; the full stack proves authenticated GETs
  always reach the worker.
- **Ordering** — `/ping` and `/ping/` answer before the wall (and
  wall when ping is off); hub upgrades 401 without a session; the
  plain-HTTP 421 dead wall.

## Measurement

No performance ledger entry: the wall makes no performance claim.
Argon2id at the `g1` constants costs tens of milliseconds and 64 MiB
per verification **by design** — that is the security parameter, not
an optimization target — and it is paid only at login, behind the
throttle and the semaphore, never on the per-request session check
(one HMAC + one map read). The structural pins are the throttle-
before-KDF ordering and the semaphore bound, both pinned by unit
tests, not benchmarks (rule 8 is satisfied by the absence of claims).

## Adversarial review outcomes

Revision 1 posed eleven open questions; the review answered every one
and added findings of its own. The rulings, all folded into the body
above:

1. **Handler-ordering slot (OQ1) — confirmed.** Between ping and hub
   is right; unauthenticated `/ping` (and `/ping/`, matching the
   handler's own table) stands. On the shared front-door port the
   mdns decider serves front-door Hosts and returns not-mine Hosts to
   the auto-HTTPS redirects — auth is inert there by construction,
   stated in Handler ordering.
2. **POST dispatch (OQ2 / DB-1) — tightened to one predicate.**
   Sign-out iff neither `user` nor `password` is present after
   parsing exactly one urlencoded body; empty-but-present values are
   failed sign-ins; non-urlencoded Content-Type → 415; duplicated
   `user`/`password`/`csrf` → 400; the 64 KB cap enforced with
   `MaxBytesReader` before `ParseForm`. Torture shapes pinned in the
   Go layer.
3. **`g1` validation (OQ3 / MF-3) — encoding pinned.**
   `base64.RawStdEncoding`: standard alphabet, no padding, exactly 64
   encoded chars → 48 bytes; `=`, `-`, `_` rejected by name at parse.
4. **Throttle keying (OQ4 / MF-2) — hardened.** IPv6 keys on the /64
   prefix; the map is bounded (65,536; expired swept first, then
   arbitrary eviction); the CGNAT residue (shared-IP lockout of a
   targeted username) is accepted for v1 and bounded by the 15-minute
   window.
5. **Reload revocation (OQ5 / MF-5) — timing moved to App Start**
   (past the reload's point of no return; an aborted reload never
   logs users out — the mdns no-rollback precedent). A user still
   present in any enabled site's set survives; global-off revokes
   everything. With per-request authorization (DB-2) revocation is
   promptness, not the boundary.
6. **Semaphore sizing (OQ6 / MF-2) — 2 confirmed, queue bounded.**
   Past 16 queued verifications a login fast-fails 429; a
   distinct-pair spray can never stack unbounded waiters. The memory
   floor (2 × 64 MiB) is stated under Login hardening.
7. **Hub upgrade 401 (OQ7) — confirmed**, with the tenant obligation
   stated: a WS client whose reconnect answers 401 sends the human to
   `/auth` itself.
8. **Session-cookie exposure (OQ8 / MF-1) — ruled: STRIP.** The
   wall's cookies are stripped from the proxied request and the hub
   snapshot; the cache bypass became the explicit auth signal
   (Sessions and Cache interplay carry the corollary in full).
9. **`Accept` detection (OQ9) — confirmed as a substring test**,
   stated as such; the WS-upgrade carve-out (never 302) closes the
   one shape that mattered.
10. **Command surface (OQ10) — ruled: `janus-auth-hash`**, namespaced,
    no generic top-level `auth` command; TTY prompt without echo,
    one-line stdin read when piped (MF-7).
11. **Plain-HTTP walls (OQ11 / DB-3) — ruled: the per-request 421
    dead wall**, not a provision/Start error — a static check cannot
    distinguish the legal `http://*.local { janus }` front-door
    carrier from a genuine plain-HTTP data-plane site and would
    reject `Caddyfile.example`. Isolated in one function; loud body +
    once-per-site ERROR; revision 1's OQ11 prose corrected in Handler
    ordering.

Review findings beyond the ledger, all adopted: per-request site-set
authorization (DB-2); the status page's re-minted CSRF pair (MF-4);
the provision/Start lifecycle split (MF-6); the sign-in retry reusing
the submitted double-submit pair within its 600 s window; the
no-`WWW-Authenticate` 401; the `GET /1.0` `"auth"` boolean; the
`auth off` remedy callout for catch-all sites; the expired-session
POST limitation stated; `HEAD /auth` setting no cookies; the `/auth`
method table beating the wall's 401; the DELETE id-prefix
unique/404/409 resolution; and the global `auth off` line following
the cache/hub precedent.
