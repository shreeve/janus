Rip Server vs Caddy — capability matrix

Exhaustive inventory of rip-lang/packages/server (v1.4.x) against Caddy 2. Includes bam-style WebSocket↔HTTP bridge and Caddy-only edge capabilities as Rip holes.

How to read: Rip-only rows are mostly the Sinatra app framework, workers, serve() bundler, and the built-in WS hub — keep those behind Caddy. Suggested split

  • Caddy at the public edge: TLS/ACME (incl. DNS-01/wildcards), HTTP/3, L7 proxy+WS upgrade, optional caddy-l4, rate limits, trusted proxies, metrics, admin API.
  • Rip behind Caddy: Sinatra API, read()/OpenAPI, sessions/CSRF, serve() bundler, worker pool, SSE watch, and the WS hub / bam-style HTTP bridge if you still want that product.
  • Custom Caddy modules fill anything Rip uniquely had at the edge that you still want in-process with Caddy (e.g. a slim realtime hub).
