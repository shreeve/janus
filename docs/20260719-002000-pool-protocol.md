# Janus ↔ Rip Server: pool coordination protocol

The contract between Janus (edge) and a tenant orchestrator (Rip
Server) for routing traffic to worker pools, including hot reload.
This is the client contract Phase 8 implements against; Phases 3–5
build the Janus half.

## Roles

| Actor | Owns | Never does |
| --- | --- | --- |
| **Janus** | TLS, host admission, `upstreams[]` per app, LB (`least_conn`), health flags, 404/503 | Spawn processes, watch files, know about pool generations, call the tenant |
| **Rip Server** | Worker spawn/kill, file watch, debounce/coalesce, pool lifecycle, sock paths, doorbell stub while dirty | Serve app code; sit on the steady-state data path |
| **Workers** | App traffic on unix sockets; private `/ready` for Rip | Talk to Janus |

Data plane, steady state: `Client → Janus → worker unix socket`. Rip
Server touches only `/1.0` — except the **doorbell** (below), a stub it
publishes as the sole upstream while the app is dirty. The client's
request is never forwarded to it: Janus **rings** it with a tiny
synthetic request, waits for ready, then proxies the untouched original
request to the fresh pool. The doorbell never sees client traffic and
never executes app code.

## The invariant

**Janus never admits a *new* request to a generation the tenant has
declared stale.** Completing an in-flight response already running on
the old generation is allowed (finishing a commitment); selecting the
old generation for a new request is not.

A file change can rename tables, change config, or break contracts —
serving new traffic from known-stale code is corruption, not uptime.

## The ring, end to end (canonical happy path)

```text
Client sends request headers
          ↓
Janus resolves app
          ↓
Doorbell selected
          ↓
Janus leaves original body unread
          ↓
Janus sends tiny bodyless ring to Rip
          ↓
Rip single-flights pool startup
          ↓
Rip publishes worker sockets          (PUT, awaits the 200)
          ↓
Rip returns internal 204              (empty, advisory)
          ↓
Janus re-resolves app
          ↓
Janus streams original request + body once
          ↓
Fresh worker responds
          ↓
Client receives the real response
```

Two steps are load-bearing: the PUT completes **before** the 204 is
sent (so the re-resolve is guaranteed to find sockets), and the final
stream is a **first delivery** (the body was never read, so nothing
replays). Non-happy branches: boot failure → `PUT []`, ring answered
`503` with the boot error, forwarded to the client; saved again
mid-boot → Rip scraps and rebuilds once from the latest files before
publishing; re-resolve finding a doorbell again → ring again, capped.

## Janus's entire model

One concept: **routable = registered host + at least one healthy
upstream.** There is no READY/NOT-READY enum, no `PREPARE` verb, and
no Janus→tenant RPC. All control flows tenant → Janus on `/1.0`.

| Registry field | Meaning |
| --- | --- |
| `id`, `hosts` | Admission: which public hosts map to this app |
| `upstreams[]` | Unix socket paths Janus may dial. **Empty = not routable**. An entry may be flagged `doorbell: true` (ring, don't forward; no health accounting) |
| heartbeat clock | Supervisor liveness — independent of `upstreams[]` |

### Control surface

| Call | Purpose |
| --- | --- |
| `POST /1.0/apps` | Register app: hosts, mint id |
| `PUT /1.0/apps/{id}/upstreams` | Atomic swap of the full sock list |
| `POST /1.0/apps/{id}/heartbeat` | Supervisor alive |
| `DELETE /1.0/apps/{id}` | Deregister |

### Write ordering (why no fencing)

The app's supervisor is the **single writer** of its upstream list, and
it **awaits each PUT's 200 before sending the next control message**.
Sequential awaited writes on one channel cannot reorder, so the
registry always holds the last thing the supervisor said, in the order
it said it. No generation numbers cross the wire; a stale-write fence
would defend against a race the protocol cannot produce.

"Is this pool already stale?" (files changed again mid-boot) is the
supervisor's **private** bookkeeping — a local flag checked before
publishing — never a protocol field.

### Data plane decision table

| Condition | Response |
| --- | --- |
| Unknown host | **404** |
| Known host, `upstreams` empty | **503** + `Retry-After` — fail fast; nothing to wake |
| Known host, doorbell upstream | Ring it; park the request (bounded hold); proxy on 204, forward its 503 on failure |
| Known host, all upstreams unhealthy | **502/503** |
| Known host, healthy upstreams | `reverse_proxy` (`least_conn`, unix) |

Janus never blocks an application request on **control-plane**
activity. The doorbell park is not that — it is a bounded wait on an
upstream's response (capped duration, capped waiters, dev-scale). An
*unbounded* edge queue across tenant reboots is the rejected shape:
arrival rate × boot time of goroutines, when the clients' own retry is
already the queue.

### Heartbeat ≠ readiness

Heartbeats prove the **supervisor** is alive, not that workers exist.
During a reload the tenant keeps heartbeating while `upstreams` is
empty — Janus treats the app as *alive but not routable*: certs stay
allowed, hub state stays, registry entry stays. Only a **missing
heartbeat** marks the app dead. One dirty app never affects another;
all state is per app id.

## Rip Server control loop

### Startup

```text
register app (hosts) → spawn pool → workers /ready
→ PUT upstreams [socks…] → begin heartbeats
```

### Dirty epoch (watch on)

Saves only **note** dirtiness; workers relaunch on **demand**, not per
save. The demand signal is the **doorbell**: a socket Rip Server itself
listens on, published as the app's only upstream while dirty. To Janus
it is just an upstream — no new edge concept.

```text
fs events → settle (trailing ~100–150ms) → content-hash gate
  bytes unchanged → no-op (format-on-save storms cost nothing)
  changed → dirty epoch:

first dirty since last publish:
    PUT upstreams [{doorbell.sock, doorbell: true}]
                                          ← admission cut: old socks gone
    OLD pool: no new work; drain grace ~2–5s → SIGTERM → wait exit → unlink
further saves while dirty:
    note it locally (dirty flag) — no Janus traffic, no spawn, nothing

request arrives at Janus → upstream is flagged doorbell:
    Janus does NOT forward the request; it RINGS: a tiny synthetic
    `GET /ring` on the doorbell socket, while the client's request
    waits in hand, body unread (client stalls on TCP backpressure)

doorbell receives a ring (someone actually wants the app):
    single-flight prepare from the current files:
        spawn NEW pool, unique sock paths (never reuse a live path)
    hold the ring open (capped ~15s, capped waiters); concurrent rings
    join the same wait — never a second cohort

prepare finishes:
    no saves since spawn began →
                    PUT upstreams [socks…], AWAIT the 200
                    (doorbell retired; registry now holds the socks)
                    → THEN answer rings 204 (empty — a pure wake-up)
                    → Janus re-reads the app's socket list from its
                      own registry, picks a worker, and proxies the
                      original request: a normal, first-time,
                      streaming delivery
    saved again mid-boot → scrap this pool, loop once for the latest files
    boot failed   → PUT upstreams []; answer rings 503 with the
                    boot error; Janus forwards that to the client; next
                    save or explicit restart retries
```

The socket list travels **only** on the PUT — never in the ring
response. The PUT is needed regardless (doorbell retirement, eager
mode has no ring, `/1.0` introspection must reflect truth), so
piggybacking socks on the 204 would just create a second registry
write path.

The 204 carries **nothing** — no generation, no socks. It is advisory:
Janus acts only on its own registry after waking, and every post-wake
state is already handled (socks → proxy; doorbell again → ring again,
capped; empty → 503). Because no state crosses the ring channel, there
is no cross-channel ordering to verify.

Properties this sequencing buys:

- **Admission cut is immediate** — the stale window closes at the
  doorbell PUT, not when processes die.
- **Saves are free** — 200 saves = one doorbell PUT, zero spawns; the
  pool relaunches once, when a request actually arrives, from the
  latest files.
- **Single-flight** — one spawn cohort per app, ever; concurrent
  arrivals coalesce onto the same wait.
- **Check at publish** — a pool whose files changed again mid-boot is
  scrapped, never published (Rip's private dirty flag), making
  serve-known-stale unreachable rather than unlikely.
- **First-request latency = one boot** — the honest price of never
  serving stale; `readyWhen: 1` keeps it to the fastest worker.

**Eager option:** `reload: eager` starts the spawn at settle instead of
waiting for the doorbell — for workflows where the post-save refresh
must feel instant and background spawns are acceptable. Default is
on-demand.

### The ring (Janus side)

The doorbell check happens at **upstream selection, before Janus has
touched the request**. So the client's request is never forwarded to
the doorbell — there is nothing to buffer and nothing to replay. The
`doorbell: true` flag tells Janus to:

1. **Ring instead of forward** — send its own tiny `GET /ring` down the
   doorbell socket. The client's request waits untouched; an in-progress
   upload simply stalls on TCP backpressure and resumes later. Rings
   for the same app share one doorbell connection outcome (the demand
   signal needs no fan-out).
2. **Wait for the ring's answer** — `204`: re-read the app's socket
   list from the registry (Rip's PUT landed before the 204 was sent,
   so the fresh socks are there), pick a worker, and proxy the original
   request as a normal, first-time, streaming delivery. The 204 is an
   empty, advisory wake-up — Janus trusts only its own registry, and
   every post-wake state has defined handling (socks / doorbell again /
   empty). `503`: forward it to the client (it carries the boot error —
   dev-visible diagnostics).
3. **Exclude the doorbell from health accounting** — ring answers are
   protocol, not sickness.
4. **Cap rings** (~3): if after a `204` the re-lookup finds a doorbell
   again (saved mid-boot), ring again; past the cap, 503 +
   `Retry-After`.
5. **Client disconnect during the hold** — abandon that waiter; the
   single-flight spawn still runs to completion (the boot is not
   wasted — the next request lands on a warm pool).

Because the request is delivered exactly once — to the fresh pool or
nowhere — there are no redirect-interception rules, no body caps, and
no idempotency questions: a held POST was never executed anywhere else.
For clients sending `Expect: 100-continue`, Janus simply withholds the
`100` until the pool is ready — the body is never even solicited early.

One operational constraint: Janus's request-read timeouts must exceed
the ring hold cap, since the client connection idles during the boot.

### In-flight requests at the dirty moment

Best-effort complete during the drain grace, then hard cut. Watch mode
therefore *can* sever a slow upload or SSE stream after grace — that is
the documented cost of editing files under an app; production runs
watch off.

### Watch hygiene

- Allowlist (`*.rip` + explicit config), never the whole tree; ignore
  `.git`, `node_modules`, caches, editor temp files.
- Rename-safe: editors write temp-then-rename; dirty on the real
  target once, not twice.

### Shutdown

`PUT upstreams [] → drain → kill → DELETE app` (or just stop
heartbeating and let the TTL reap it).

## Defaults

| Knob | Dev (watch on) | Prod |
| --- | --- | --- |
| watch | ON | OFF; enabling requires an explicit flag and logs loudly |
| `w` | 2 | sized for RPS |
| `c` | 1 | 1; raise only with watch off |
| settle | 100–150 ms trailing | — |
| reload trigger | on-demand (doorbell); `eager` opt-in | — |
| doorbell hold | ~15 s ring cap, capped waiters, then 503 | — |
| ring retries | 3 (re-dirty mid-boot), then 503 + `Retry-After` | — |
| ready gate | ≥1 worker → PUT socks, rest join the list as they boot | all `w` before PUT |
| drain grace | 2–5 s | — |
| unroutable clients | 503 + `Retry-After` | — |

Steady state (watch off): no watcher exists, no dirty machinery is
live, and the Janus hot path is host lookup → atomic upstream pointer
→ proxy. No gen comparisons, no registry locks, no tenant calls per
request.

## Rejected designs

| Rejected | Why |
| --- | --- |
| Keep old socks selectable across saves ("zero-downtime" flip) | New requests reach known-stale code — the exact corruption the invariant forbids |
| Janus `PREPARE` RPC to the tenant | Inverts the control arrow; makes the edge an app orchestrator; second state machine duplicating "no upstreams" |
| Eager respawn on every save | Burns a pool boot per save burst with no one asking; watch is a dev posture where saves vastly outnumber requests. The doorbell keeps saves free and rebuilds once, on demand |
| Lazy rebuild with **no** demand signal (bare `[]` + Janus 503) | Rip never learns a request arrived; app stays down until something unrelated wakes it. The doorbell is the demand signal without a PREPARE verb |
| **Janus** holds requests until the pool is ready | Unbounded queue amplification at the edge during boot. Holding lives in Rip's doorbell only — capped duration, capped waiters, dev-scale |
| Doorbell 307s to the **client** | Non-browser clients (curl, webhooks, APIs) don't follow redirects; extra RTT; visible protocol noise |
| Forward the client's request to the doorbell, buffer, replay on a marked 307 | Wasted work: the doorbell flag is known at upstream selection, so Janus can ring with a synthetic request and keep the client's request untouched — no buffering, no body cap, no replay/interception rules. The request is delivered exactly once, to the fresh pool |
| READY/NOT-READY enum in the registry | Redundant with empty vs non-empty `upstreams[]`; version-skew magnet |
| macOS `SO_REUSEPORT` worker LB | Sticky last-binder; paused listener still receives SYNs (verified live, Darwin 25, 2026-07) |
