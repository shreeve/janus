# Janus ↔ Rip Server: pool coordination protocol

The contract between Janus (edge) and a tenant orchestrator (Rip
Server) for routing traffic to worker pools, including hot reload.
This is the master description of the protocol: the build SPEC's
Phases 3–5 build the Janus half against it, and Phase 8's tenant
implements the Rip Server half. Anything that contradicts this file
is stale.

## Roles

| Actor | Owns | Never does |
| --- | --- | --- |
| **Janus** | TLS, host admission, `upstreams[]` per app, load balancing (`least_conn`), health flags, 404/503 | Spawn processes, watch files, track pool generations, initiate control-plane calls to the tenant |
| **Rip Server** | Worker spawn/kill, file watch, debounce/coalesce, pool lifecycle, socket paths, the doorbell | Serve app code; sit on the steady-state data path |
| **Workers** | App traffic on unix sockets; answer `GET /ready` for Rip Server | Talk to Janus |

Data plane, steady state: `Client → Janus → worker unix socket`.
Rip Server talks to Janus only on `/1.0`, with one exception: the
**doorbell**, a wake-up socket Rip Server owns, published as an app's
only upstream during reload (see *The ring*). The doorbell never sees
client traffic and never executes app code.

## The invariant

**Janus never admits a *new* request to a generation the tenant has
declared stale.** (A *generation* is one spawned cohort of workers;
the term is conceptual — it never appears in the protocol.) Completing
an in-flight response already running on the old generation is allowed
— finishing a commitment. Selecting the old generation for a new
request is not.

A file change can rename tables, change config, or break contracts —
serving new traffic from known-stale code is corruption, not uptime.

## Janus's model

One concept: **routable = registered host + at least one healthy
upstream.** There is no READY/NOT-READY enum, no `PREPARE` verb, and
no Janus-initiated control call to the tenant. All control flows
tenant → Janus on `/1.0`.

| Registry field | Meaning |
| --- | --- |
| `id`, `hosts` | Admission: which public hosts map to this app |
| `upstreams[]` | Unix socket paths Janus may dial. **Empty = not routable** |
| heartbeat clock | Supervising process is alive — independent of `upstreams[]` |
| cache generation | Per-app atomic counter, bumped in the same critical section as every registry write that purges the cache; read by cache fills before host resolution ([cache capability](20260720-033201-capability-microcache.md)) |

### Control surface

All calls are JSON over the control listeners configured by the
`control` capability (internal / local / public, Bearer token when
set); there is no additional per-call auth in this protocol. Unknown
`{id}` → 404; malformed body → 400.

| Call | Body → Response |
| --- | --- |
| `POST /1.0/apps` | `{"name":"shop","hosts":["shop.example.com"]}` → `201 {"id":"shop-x7k2p9"}`; host already claimed → 409 (first-wins) |
| `PUT /1.0/apps/{id}/upstreams` | `{"upstreams":[{"path":"/run/shop/w1.sock"}, …]}` → `200`. Atomic full-list swap. An entry may carry `"doorbell":true` — a doorbell entry must be the **only** entry; a list mixing doorbell and worker entries, or containing more than one doorbell, → 400 |
| `POST /1.0/apps/{id}/heartbeat` | empty → `204` |
| `DELETE /1.0/apps/{id}` | → `204` |

Any upstreams PUT (and `DELETE`, TTL reap, and host claim via
`POST /apps`) atomically drops the app's cache entries **and bumps the
app's cache generation** — the admission cut also cuts the cache, and
in-flight fills that straddle the cut can neither store nor retain
waiters ([cache capability](20260720-033201-capability-microcache.md)).
`DELETE` and TTL reap additionally tear down the app's hub (closing its
WebSocket connections); upstreams PUTs never touch hub state.

The registry is memory-only. If Janus restarts, apps vanish; Rip
Server notices (heartbeat → 404) and re-registers, then re-publishes
its current upstream list. On a PUT/heartbeat failure or timeout,
Rip Server retries with backoff; being the single writer, it sends no
later control message until the retry resolves.

### Write ordering (why no fencing)

Rip Server is the **single writer** of its app's upstream list, and it
**awaits each PUT's 200 before sending the next control message**.
Because each write is acknowledged before the next is sent, the
registry always holds the writer's most recent statement. No
generation numbers cross the wire; a stale-write fence would defend
against a race the protocol cannot produce.

Whether a booting pool is already stale (files changed again mid-boot)
is Rip Server's private bookkeeping — a local dirty flag checked
before publishing — never a protocol field.

### Data plane decision table

Checked top to bottom per request:

| Condition | Response |
| --- | --- |
| Unknown host | **404** |
| Known host, cache HIT (site cache on, no bypass, app id and variant match) | Serve from memory; no upstream selected, nothing counted toward health ([cache capability](20260720-033201-capability-microcache.md)) |
| Known host, `upstreams` empty | **503** + `Retry-After` — down on purpose; nothing to wake |
| Known host, doorbell upstream | Ring it; hold the request (bounded); proxy on 204, forward its 503 on failure |
| Known host, all upstreams unhealthy | **503** (+ `Retry-After`); **502** only when a dial succeeded and the worker misbehaved |
| Known host, healthy upstreams | `reverse_proxy` (`least_conn`, unix) |

Health is passive: a failed dial, a transport failure after the dial
succeeded, or a worker dying mid-response (the body copy fails after
response headers already reached the client) marks that upstream
unhealthy for a short window; no active probes. Doorbell entries are
excluded — their responses never count toward health.

**Worker-marked 503s are flow control, not failures.** A worker at its
concurrency cap answers `503` with `Rip-Worker-Busy: 1`; a draining
worker answers `503` with `Rip-Worker-Draining: 1`. Janus excludes
marked 503s from health accounting, and when the request is replayable
it immediately tries the next upstream instead of forwarding the
bounce. Replayable means Janus can deliver the request again without
double delivery: the request carries no body at all (zero
`Content-Length`, nothing chunked), or it carries `GetBody` — a
Janus-buffered request (hub bridge POSTs) that reconstructs a fresh
body per attempt. An ordinary client request whose body streams to the
worker is never replayed — the stream cannot be rewound — so its
marked 503 is forwarded to the client with the marker headers
stripped. When every upstream bounces a replayable request, the client
gets `503` + `Retry-After` — capacity, not failure. A 503 **without**
a marker is an application response and passes through verbatim, never
counted toward health and never retried.

Janus never blocks an application request on control-plane activity.
The doorbell hold is not that — it is a bounded wait on an upstream's
response (capped duration, capped waiters). The rejected shape is an
unbounded edge queue: one held goroutine per arrival for the whole
boot, duplicating the retry queue clients already provide.

### Heartbeat ≠ readiness

Heartbeats prove the supervising Rip Server process is alive, not that
workers exist. They begin **immediately after registration** and
continue through reloads and boots — an app with empty upstreams and
fresh heartbeats is *alive but not routable*: certs stay allowed, the
registry entry stays. Only a missing heartbeat (TTL expiry) marks the
app dead, with the same effect as `DELETE`: hosts → 404, and the
tenant must re-register. One dirty app never affects another; all
state is per app id.

## The ring, end to end

The canonical happy path for a request that arrives while an app is
dirty:

```text
Client sends request headers
          ↓
Janus resolves app
          ↓
Doorbell selected
          ↓
Janus leaves original body unread
          ↓
Janus sends tiny bodyless ring to Rip Server
          ↓
Rip Server starts (or joins) the one in-progress pool boot
          ↓
Rip Server publishes worker sockets      (PUT, awaits the 200)
          ↓
Rip Server returns internal 204          (empty, advisory)
          ↓
Janus re-resolves app
          ↓
Janus streams original request + body once
          ↓
Fresh worker responds
          ↓
Client receives the real response
```

Two steps are load-bearing:

- The PUT completes **before** the 204 is sent, so the re-resolve is
  guaranteed to find sockets.
- The final stream is a **first delivery** — the body was never read,
  so nothing replays.

Non-happy branches:

- **Boot failure** — the doorbell stays published; rings are answered
  503 carrying the boot error, which Janus forwards to the client.
- **Saved again mid-boot** — Rip Server scraps the pool and boots
  again from the latest files before publishing.
- **Re-resolve finds a doorbell again** — Janus rings again, capped.

## The ring (Janus side)

The doorbell check happens at **upstream selection, before Janus has
touched the request** — so the client's request is never forwarded to
the doorbell, and there is nothing to buffer and nothing to replay.
When the selected upstream is a doorbell, Janus:

1. **Rings instead of forwarding** — sends its own
   `GET /ring HTTP/1.1` (no body, `Host:` = app id) on a connection to
   the doorbell socket. The client's request holds untouched; an
   in-progress upload stalls on TCP backpressure and resumes later.
   For `Expect: 100-continue` clients, Janus withholds the `100` until
   the pool is ready — the body is never solicited early.
2. **Maintains at most one outstanding ring per app** — all requests
   holding for that app await the one ring's outcome. (Rip Server
   additionally single-flights on its side, so a stray extra ring —
   e.g. after a Janus restart — is harmless.)
3. **Caps holders per app** (waiter cap): requests beyond the cap get
   an immediate 503 + `Retry-After` instead of holding.
4. **Waits for the ring's answer, with its own timeout** (hold cap +
   margin — a dead doorbell must not hold requests forever):
   - `204` → re-read the app's socket list from the registry, pick a
     worker, proxy the held request as a normal first-time streaming
     delivery. (The 204 is empty and advisory; Janus trusts only its
     own registry. Every post-wake state is handled: sockets → proxy;
     doorbell again → ring again; empty → 503.)
   - `503` → forward it to the client; it carries the boot error.
   - connection error / timeout / EOF → treat as 503 to all holders.
5. **Caps ring retries per held request** (~3): if a re-resolve finds
   a doorbell again, ring again; past the cap, 503 + `Retry-After`.
   The counter lives in the data-plane handler holding that request.
6. **Excludes the doorbell from health accounting.**

Because the request is delivered exactly once — to the fresh pool or
nowhere — a held POST was never executed anywhere else, so there are
no idempotency questions. If the client disconnects during the hold,
Janus abandons that holder; the boot continues (the next request lands
on a warm pool).

One operational constraint: Janus's request-read timeouts must exceed
the hold cap, since the client connection idles during the boot.

## Rip Server control loop

### The doorbell socket

One per app, owned by Rip Server, created at startup, persistent
across reload epochs. It appears in `upstreams[]` only while the app
is dirty or booting. It answers `GET /ring` per this protocol and
404s anything else, immediately.

### Startup

```text
POST /apps (hosts) → begin heartbeats
watch on:  PUT upstreams [doorbell] → spawn pool
           (requests arriving mid-boot ring and hold, as in reload)
watch off: spawn pool → all workers ready → PUT upstreams [sockets…]
           (boot failure at startup = exit nonzero with the error)
```

Heartbeats start at registration — before any worker exists — so a
slow cold boot is never mistaken for a dead app.

Workers signal readiness by answering `GET /ready` with 200 on their
app socket; Rip Server polls until ready or a per-worker **boot
deadline** expires (deadline expiry = boot failure).

### Dirty epoch (watch on)

Saves only **note** dirtiness; workers relaunch on **demand** — the
ring is the demand signal. Rip Server's side:

```text
fs events → settle (trailing ~100–150ms) → content-hash gate
  bytes unchanged → no-op (format-on-save storms cost nothing)
  changed → dirty epoch:

first dirty since the last upstreams PUT:
    PUT upstreams [doorbell]        ← admission cut: old sockets gone
    OLD pool: retire (drain, below)
further saves while dirty:
    set the local dirty flag — no Janus traffic, no spawn, nothing

ring received (someone actually wants the app):
    single-flight: start one boot from the current files, or join the
    boot already in progress — never a second cohort
    spawn NEW pool on fresh socket paths (never reuse a live path)
    hold the ring open, up to the hold cap (~15s) → then answer 503

boot finishes:
    dirty flag clear → PUT upstreams [sockets…], AWAIT the 200,
                       THEN answer held rings 204
    dirty flag set   → scrap this pool (never publish it), clear the
                       flag, boot again from the latest files; repeat
                       while saves keep landing — each waiting client
                       is still bounded by its own hold cap
    boot failed      → keep the doorbell published; cache the error;
                       answer held (and future) rings 503 + the boot
                       error immediately, with NO respawn — the next
                       content-changing save clears the failure and
                       the next ring boots again
```

Publishing means any `PUT …/upstreams`. With `readyWhen: 1` (dev
default), the first ready worker triggers the sockets PUT and each
late worker triggers one more awaited full-list PUT; the dirty check
applies to every publish.

Properties this sequencing buys:

- **Admission cut is immediate** — the stale window closes at the
  doorbell PUT, not when processes die.
- **Saves are free** — 200 saves = one doorbell PUT, zero spawns; the
  pool boots once, when a request actually arrives, from the latest
  files.
- **Single-flight** — at most one boot per app at a time; concurrent
  demand joins it.
- **Check at publish** — a pool whose files changed mid-boot is
  scrapped, never published, so serving known-stale code is
  unreachable, not merely unlikely.
- **First-request latency = one boot** — the honest price of never
  serving stale; `readyWhen: 1` keeps it to the fastest worker.

**Eager option:** `reload: eager` changes only the boot trigger — the
spawn starts at settle instead of waiting for a ring. Everything else
is identical: the doorbell PUT still cuts admission first, boots are
still single-flighted, and mid-boot saves still scrap-at-publish. For
workflows where the post-save refresh must feel instant and background
spawns are acceptable. Default is on-demand.

The socket list travels **only** on the PUT — never in the ring
response. The PUT is needed regardless: it retires the doorbell, it is
the only publish path in eager mode, and `/1.0` introspection must
reflect truth. Piggybacking sockets on the 204 would only create a
second registry write path.

### Retiring the old pool

After the doorbell PUT, Janus sends the old workers nothing new. They
get a flat **drain grace** (~2–5s — no completion signal, just a
timer) to finish in-flight responses, then SIGTERM, then SIGKILL a few
seconds later if needed; socket paths are unlinked after exit. Watch
mode therefore *can* sever a slow upload or SSE stream at the cut —
that is the documented cost of editing files under a running app;
production runs watch off.

### Watch hygiene

- Allowlist (`*.rip` + explicit config), never the whole tree; ignore
  `.git`, `node_modules`, caches, editor temp files.
- Rename-safe: editors write temp-then-rename; dirty on the real
  target once, not twice.

### Shutdown

`PUT upstreams [] → drain → kill → DELETE app` (or stop heartbeating
and let the TTL reap it). The bare empty list means "down on purpose"
— it is never the reload or failure state, so Janus knows there is
nothing to wake.

## Defaults

`w` = worker processes per app; `c` = concurrent requests per worker;
`readyWhen` = how many workers must be ready before the first sockets
PUT.

| Knob | Dev (watch on) | Prod |
| --- | --- | --- |
| watch | ON | OFF; enabling requires an explicit flag and logs loudly |
| `w` | 2 | sized for load |
| `c` | 1 | 1; raise only with watch off |
| settle | 100–150 ms trailing | — |
| boot trigger | on-demand (ring); `eager` opt-in | — |
| hold cap | ~15 s per ring, then 503 | — |
| waiter cap | ~64 held requests per app, then 503 | — |
| ring retries | 3 per held request, then 503 + `Retry-After` | — |
| `readyWhen` | 1 (rest join as they boot) | all `w` |
| boot deadline | ~30 s per worker | — |
| drain grace | 2–5 s, then SIGTERM, SIGKILL ~5 s later | — |
| heartbeat / TTL | 5 s / 15 s | 5 s / 15 s |

Steady state (watch off): no watcher exists, no dirty machinery is
live, and the Janus hot path is a host lookup, an atomic read of the
upstream list, and the proxy. No generation comparisons, no registry
locks, no tenant calls per request.

## Rejected designs

| Rejected | Why |
| --- | --- |
| Keep old sockets selectable across saves ("zero-downtime" flip) | New requests reach known-stale code — the exact corruption the invariant forbids |
| Janus `PREPARE` RPC to the tenant | Inverts the control arrow; makes the edge an app orchestrator; second state machine duplicating "no upstreams" |
| Eager respawn on every save | Burns a pool boot per save burst with no one asking; watch is a dev posture where saves vastly outnumber requests. The doorbell keeps saves free and boots once, on demand |
| Lazy rebuild with **no** demand signal (bare `[]` + Janus 503) | Rip Server never learns a request arrived; the app stays down until something unrelated wakes it. The doorbell is the demand signal without a `PREPARE` verb — which is also why boot failure keeps the doorbell published instead of clearing to `[]` |
| **Janus** holds requests until the pool is ready, unbounded | Unbounded queue amplification at the edge during boot. Holding is bounded — hold cap, waiter cap — and lives behind the doorbell |
| Doorbell 307s to the **client** | Non-browser clients (curl, webhooks, APIs) don't follow redirects; extra round trip; visible protocol noise |
| Forward the client's request to the doorbell, buffer it, replay on a marked 307 | Wasted work: the doorbell flag is known at upstream selection, so Janus can ring with a synthetic request and keep the client's request untouched — no buffering, no body caps, no replay or redirect-interception rules. The request is delivered exactly once, to the fresh pool |
| READY/NOT-READY enum in the registry | Duplicates empty-vs-non-empty `upstreams[]` and invites Janus/tenant version skew |
| Generation numbers on the wire (fenced PUTs) | Defends against write reordering the protocol cannot produce: one writer, each PUT acknowledged before the next |
| macOS `SO_REUSEPORT` worker LB | The last binder receives all connections, and a paused listener still accepts SYNs (verified live, Darwin 25, 2026-07) |
