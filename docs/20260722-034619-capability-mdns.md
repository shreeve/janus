# Capability: mdns (LAN presence — `janus.local`, per-app `.local` names, the status front door)

> **The written contract for capability 5 — revision 1, awaiting
> adversarial review.** Nothing below is implemented. The core design
> decisions (process-wide scope, off by default, `brutella/dnssd`,
> per-app `.local` advertising from hot registry data, the read-only
> embedded status page, the canonical redirect + diagnostic mode, the
> HTTPS-everywhere tier as configuration-not-code) are owner-DECIDED
> and are not reopened here. Everything this revision had to pin
> beyond those decisions is collected in "Open questions for
> adversarial review" at the end, so the review knows exactly where to
> aim. Library claims below are verified against
> `github.com/brutella/dnssd` v1.2.x source (`service.go`,
> `responder.go`, `probe.go`), and one settled-design phrase is
> corrected against that source — flagged in place and in the ledger.

Cold Caddyfile capability. When enabled, Janus announces itself on the
local network over multicast DNS: it advertises a configurable
`janus.local` name, automatically advertises every registered app host
that fits the mDNS host model, and serves a plain-HTTP front door at
the advertised name — an embedded, read-only status page showing the
registry, data plane, cache, and hub at a glance. An optional
`canonical` name turns the front door into a hand-off ramp to a real
HTTPS name, with a built-in diagnostic page for the one failure mode
(DNS-rebinding filters) that otherwise presents as a mystery timeout.

| --- | --- |
| **Order** | **5** (after **ping**, **control**, **cache**, **hub**) |
| **Surface** | Cold config (Caddyfile) for the capability; hot registry data drives per-app advertising; advertiser state on hot `/1.0/mdns` |
| **Cascades** | **No — process-wide only** (like **control**; a site-level occurrence is a parse error) |
| **Built-in default** | off (no directive → no multicast, no front door, no new listener) |
| **Module** | `janus` app (`mdns.go`, `mdns_config.go`, embedded `mdns.html`; control surface in `control_mdns.go` per the `control_` prefix ruling) |
| **Directive** | Global options: `janus { mdns … }` |
| **Library** | `github.com/brutella/dnssd` — maintained, passes Apple's Bonjour Conformance Test; probing, conflict resolution, goodbye packets, multi-interface, IPv6 |
| **Status** | **Contract — revision 1**; adversarial review pending |

## Why it exists

Janus's public face assumes DNS: real names, SNI-selected sites,
publicly trusted certificates. On a bare LAN — a dev machine, a home
lab, a workshop network — none of that exists, and the first-run
experience is "find the box's IP, type it with a port, get a
certificate warning." mDNS closes that gap with zero configuration on
the client side:

- **`janus.local` answers immediately** on every current OS (macOS,
  iOS, Windows 10+, Linux with Avahi, Android 12+) with no DNS server,
  no router config, no client install. The A/AAAA record is the
  load-bearing part — Safari and friends resolve the name and connect;
  the accompanying `_http._tcp` service advertisement is a bonus for
  browse-based discovery.
- **Registered apps get LAN names as a side effect.** A tenant that
  registers `hosts: ["myapp.local"]` is advertising to the LAN the
  moment the registration lands — hot registry data driving hot
  behavior, the same shape as `/1.0/tls/ask` allowance following the
  registry lifecycle. The heartbeat reaper removing an app withdraws
  its names the same way.
- **The front door explains the system.** The status page answers
  "what is this box running?" without the operator teaching anyone
  curl or handing out control-plane access: apps, hosts, worker
  health, heartbeat freshness, cache and hub counters — strictly
  read-only, socket paths redacted.
- **The `canonical` option graduates the LAN to real HTTPS.** The
  HTTPS-everywhere tier (below) is pure configuration on stock Caddy
  modules; the front door's redirect + diagnostic mode is the one
  piece that needs code, because the failure mode it catches —
  router DNS-rebinding filters eating a public name that resolves to
  a private address — is invisible from the server side and
  maddening from the client side.

Default **off**: enabling mdns announces this machine and its app
list to everyone on the local network. That is the product being
bought, and it is opt-in.

## Scope

| Piece | Where | Role |
| --- | --- | --- |
| `janus { mdns … }` | Global options `{ }` | Process-wide: one advertiser, one front door |
| `janus` | Site block | Data-plane admission, unchanged — **`mdns` is illegal here** |

One process, one LAN identity. A site-level `janus { mdns … }` is a
parse error, exactly as site-level `control` is.

**Advertising is not admission.** mDNS makes a name *resolvable*; the
cold admission gate decides whether Janus *serves* it (rule 2: hot
JSON never changes the cold gate). A registered `myapp.local` is
advertised the moment it is registered, but a browser reaching it gets
a page only if some cold site block admits that host — a catchall
site, an explicit block, or the on-demand-TLS site, same as any other
registered host. The status page is the one exception by design: it is
served by the front-door listener the capability itself provisions
(the way **control** provisions `/1.0` listeners), so `janus.local`
works with no site block at all.

## Advertising mechanism (verified against the library)

`brutella/dnssd`'s unit of registration is a **service**
(`dnssd.Config{Name, Type, Domain, Host, Port, Ifaces}`), not a bare
hostname — the library has no standalone "advertise this A record"
API. The responder answers A/AAAA queries for the `Host` of every
registered service, probes for hostname and instance-name uniqueness
before announcing (RFC 6762 §8), and withdraws records with TTL-0
goodbyes on removal. So each advertised name is carried by exactly one
service registration:

| Name | Service registration | Why this shape |
| --- | --- | --- |
| The configured name (default `janus.local`) | `Type: _http._tcp`, `Port: 80`, `Host: <label>` | The front door is plain HTTP on 80; the browsable URL is `http://janus.local/` with no port |
| Each advertisable registered app host | `Type: _https._tcp`, `Port: 443`, `Host: <label>` | App sites are TLS-terminated by Caddy on 443 |

The A/AAAA answering — the load-bearing part — rides on those
registrations: a hostname query for `janus.local.` is answered with
the machine's own interface addresses, exactly what a proxy-host
`dns-sd -P` advertisement achieves, but from inside the process with
probing and goodbyes handled. (The settled design's phrase
"proxy-host style" is corrected here to the actual mechanism: the
library's deprecated `IPs` field is not used; addresses derive from
interfaces, below.)

**Advertisable app hosts.** A registered host is advertised iff it has
the mDNS host shape: exactly one label plus `.local`
(`myapp.local` yes; `myapp.dev.local` no — multi-label `.local` names
are outside the RFC 6762 host model the library implements;
`myapp.ripdev.io` no — non-`.local` names are never multicast).
Non-advertisable hosts are skipped and **counted**
(`skipped_hosts` on `/1.0/mdns`), and each skipped `.local` host is
logged once per registration — visible, never silent, but not an
error: registration is hot-plane and must not fail because cold mdns
cannot carry one of its names. A registered app host equal to the
configured name coexists with it (two service registrations sharing
one hostname is legal DNS-SD).

**Addresses and interfaces.** Addresses are derived from interfaces at
answer time, never configured:

- `interface <name>…` unset (the default): the advertiser
  auto-selects every interface that is **up, multicast-capable,
  non-loopback, and holds at least one usable unicast address**; it
  passes the library `BlockedIPNets` of `127.0.0.0/8`, `::1/128`, and
  `169.254.0.0/16` so loopback and IPv4 link-local addresses never
  appear in answers. IPv6 link-local (`fe80::/10`) is legitimate mDNS
  material and is advertised. A and AAAA both, per interface.
- `interface en0` (one or more names): pin to exactly those
  interfaces. A pinned interface that does not exist at Start is a
  hard error — the operator named it; absence is misconfiguration.

**Conflict resolution.** Probing detects a name already claimed on the
LAN; the library resolves per RFC 6762 by renaming
(`janus.local` → `janus-2.local`). Janus cannot reject a runtime LAN
condition at parse time, so the contract is loudness: the effective
advertised name is logged at WARN when it differs from the configured
name, and both configured and effective names appear on `/1.0/mdns`
and the status page. A second Janus on the same LAN is therefore
visible, never mysterious.

**IP address changes — stated honestly.** The library computes A/AAAA
answers from the interfaces' *live* addresses on every query, so a new
query after a DHCP change gets the new address immediately. What the
library does **not** do is proactively re-announce on address change
(hot-plug support is an open item in its own TODO). The v1 contract is
therefore: **query-time freshness, no proactive re-announcement** —
a peer holding a cached record keeps it for at most the record TTL
(RFC 6762 prescribes 120 seconds for host address records). This
deviates from the settled design's "IP change re-announces" phrase and
is flagged in the ledger; a proactive interface-watcher is listed
under Future work.

**Asynchrony is contractual.** No registry mutation ever performs
multicast I/O in its critical section. The registry gains one
notification hook in the existing `purge`/`hubTeardown` pattern —
invoked outside the lock on create, PATCH (host changes), DELETE, and
TTL reap — and the advertiser owns a single reconcile goroutine that
converges the responder's registration set to the desired set
(configured name ∪ advertisable registered hosts). Registration and
reap latency are untouched; advertising converges within one
reconcile pass. Probing (~1s per new name, per the RFC) happens on the
advertiser's goroutine, never on a control-plane request.

## The front door

When mdns is enabled, the janus app opens one plain-HTTP listener —
the way **control** opens `/1.0` listeners — and serves the status
page on it. No site block, no TLS, no auth.

| | |
| --- | --- |
| Listen | `http://0.0.0.0:80/` — fixed in v1 (the browsable URL must be portless) |
| Routes | `GET /` → the embedded status page; `GET /status.json` → the snapshot; `HEAD` allowed on both |
| Everything else | Unknown path → **404**; known path with another method → **405** — the dashboard has zero mutating endpoints, enforced by routing, not convention |
| Host header | Not checked: the listener is the identity (the control-listener model). `http://<lan-ip>/` works, which the diagnostic story wants |
| Caching | `Cache-Control: no-store` on both routes |

**Bind failures are hard Start errors.** If `:80` is held by another
process, Start refuses the config with a precise error naming the
address and the conflict. If the *same config's* HTTP app has a server
listening on `:80` (e.g. auto-HTTPS redirects, an explicit `http://`
site), Start refuses with an error naming the colliding server and the
two remedies (`auto_https disable_redirects`, or move/remove the
site) — Caddy's listener pooling would otherwise share the socket
between two servers and split requests arbitrarily, which is exactly
the silent drift the rules forbid. The check runs at Start, where the
HTTP app is provisioned (the hub's site-table precedent).

### The status page

One embedded HTML file (`mdns.html`, `go:embed`), hand-written,
**zero external resources** — no CDN, no framework, no build step. The
offline-LAN story ("no internet, `.local` still works") is only true
if the page carries everything it needs. The v3 `server.html`
dashboard is the tonal ancestor (status cards, poll refresh, deep
links); its CDN-loaded Tailwind/Alpine stack deliberately does not
port.

The page fetches `/status.json` on load and every **10 seconds**
(fixed in v1), and renders:

| Section | Content | Source (all in-process — never a `/1.0` proxy) |
| --- | --- | --- |
| Apps | Name, id, hosts (each a `https://<host>/` deep link), upstream **count** and **healthy count**, heartbeat freshness ("2s ago"), advertised badge on `.local` hosts | Registry snapshot + data-plane health reads |
| Cache | Process hit/miss/coalesce/bypass counters; per-app hits | The existing `cacheStore` snapshot |
| Hub | Process and per-app connection and channel counts | The existing hub counters |
| mDNS | Configured name, effective (post-conflict) name, advertised names, skipped-host count, canonical target when set | Advertiser state |

`GET /status.json` returns exactly that data:

```json
{
  "name": "janus.local",
  "effective_name": "janus.local",
  "canonical": "https://janus.lan.ripdev.io",
  "advertised": ["janus.local", "shop.local"],
  "skipped_hosts": 0,
  "apps": [
    {
      "id": "shop-x7k2p9",
      "name": "shop",
      "hosts": ["shop.ripdev.io", "shop.local"],
      "upstreams": { "total": 4, "healthy": 4 },
      "heartbeat_age_ms": 1200,
      "cache": { "hits": 0, "misses": 0, "coalesced": 0 },
      "hub": { "conns": 0, "channels": 0 }
    }
  ],
  "cache": { "hits": 0, "misses": 0, "coalesced": 0, "bypass": 0 },
  "hub": { "conns": 0, "channels": 0 }
}
```

**Redaction is contractual, not cosmetic:** upstream socket paths
appear nowhere in `/status.json` or the rendered page — counts and
health only. `bridge_path` is likewise omitted (tenant internals, same
class as socket paths). The snapshot is allowlist-shaped: only the
fields above are emitted, so a future registry field is private on
this surface until someone deliberately adds it. An acceptance case
registers an app with a distinctive socket path and asserts those
bytes are absent from the entire response.

**Heartbeat freshness needs one small registry addition, stated
explicitly:** `heartbeatAt` is unexported and absent from the
registry's JSON today (`apps.go`), by design. The registry gains an
in-process, read-lock-only accessor (per-app age since last heartbeat)
consumed by the status snapshot. The `/1.0/apps` JSON representation
is **unchanged** — heartbeat freshness is exposed on this read-only
LAN surface and on `/1.0/mdns`'s owning process only as
`heartbeat_age_ms`, never as a wall-clock timestamp. The data plane
likewise gains a read-only per-path "currently marked unhealthy"
snapshot (an atomic read of the existing `unhealthyUntil`); neither
addition touches a hot path.

### Redirect + diagnostic mode (`canonical`)

Unset (the default), the front door is the dashboard and nothing else.
Set — `canonical https://janus.lan.ripdev.io` — the front door
becomes the hand-off ramp:

1. On load, the page probes the canonical origin **client-side**: a
   `no-cors` fetch of the canonical origin with a short (~3s) timeout.
   Any completed response — any status, opaque or not — proves
   DNS + TCP + TLS reachability *from this client*, which is the only
   vantage point that matters and the reason this cannot be a
   server-side 3xx: the server has no way to know whether the client's
   resolver filters the name.
2. **Probe succeeds** → the page redirects
   (`location.replace(canonical)`).
3. **Probe fails** → the page renders a one-screen explanation
   instead of letting the user discover a mystery timeout later: what
   was tried, why it likely failed (the canonical name resolves to a
   private LAN address, and many routers' DNS-rebinding protection
   silently refuses exactly that), and the router-specific one-line
   fix (table below, embedded in the page). The dashboard renders
   beneath the explanation — diagnostic mode never costs the
   information the front door exists to show.
4. **Loop guard:** the page skips the probe and redirect when
   `location.origin` already equals the canonical origin — required
   because the worked pattern below serves this same page *at* the
   canonical name.

## Cold config

### Syntax

Normal Caddyfile grammar. The directive takes no arguments — presence
enables; absence is off; there is no `mdns on|off` form (nothing
cascades, so there is nothing for `off` to beat).

```caddyfile
{
	janus {
		control local
		ping
		mdns                    # minimal: advertise janus.local + .local app
		                        # hosts, front door on http://0.0.0.0:80/
	}
}
```

```caddyfile
{
	janus {
		control local
		mdns {
			name janus.local                        # default; <label>.local only
			canonical https://janus.lan.ripdev.io   # optional: redirect + diagnostic mode
			interface en0                           # optional pinning; default auto-select
			apps off                                # suppress per-app .local advertising
		}
	}
}
```

### Legal lines

```text
mdns
mdns {
	name <label>.local          # exactly one arg; lowercased; single label + .local
	canonical <https-url>       # exactly one arg; https:// origin, port allowed
	interface <name>…           # one or more interface names
	apps on|off                 # per-app .local advertising (default on)
}
```

### Defaults

| Knob | Default |
| --- | --- |
| capability | off (directive absent) |
| `name` | `janus.local` |
| `canonical` | unset — plain dashboard, no probe, no redirect |
| `interface` | auto-select: every up, multicast-capable, non-loopback interface with a usable unicast address; A + AAAA both; `127.0.0.0/8`, `::1/128`, `169.254.0.0/16` never advertised |
| `apps` | on |
| Front-door listen | `http://0.0.0.0:80/` (fixed in v1) |
| Status poll interval | 10s (fixed in v1) |
| Canonical probe timeout | ~3s (fixed in v1) |
| Record TTLs | The library's RFC 6762 defaults (host address records 120s) |

### Hard errors (reject at parse)

- `mdns` with any argument (`mdns on`, `mdns off`, anything — presence
  is the switch)
- Duplicate `mdns` directive in the global block
- Unknown subdirective inside the block; duplicate subdirective
- Nested block under any subdirective
- `name` without exactly one argument
- `name` not of the form `<label>.local`: bare label (`janus`),
  multi-label (`a.b.local`), wrong suffix (`janus.lan`), or an illegal
  label (character set, leading/trailing hyphen, > 63 bytes — the
  registry's host-label rule reused). Uppercase is lowercased
  (hostnames are case-insensitive), never rejected
- `canonical` without exactly one argument
- `canonical` not an `https://` URL; carrying a path, query, fragment,
  or userinfo (it names an origin, not a page); missing or implausible
  hostname (registry host validation reused); or naming a `.local`
  host (redirecting `.local` to `.local` defeats the mode's purpose)
- `interface` with zero arguments; an empty or duplicate interface
  name
- `apps` with anything but exactly `on` or `off`
- Site-level `janus { mdns … }` (process-wide only — the `control`
  rejection, verbatim shape)

### Hard errors (refuse at provision / Start)

- A pinned `interface` name that does not exist on this machine
- Front-door bind failure (address held by another process), named
  precisely
- Front-door address collision with an HTTP-app server in the same
  config (auto-HTTPS redirect listener, explicit `http://` site) —
  the error names the colliding server and both remedies

## Behavior contract

### Advertising

| Row | Contract |
| --- | --- |
| Enable | `mdns` present → the configured name probes, announces, and answers A/AAAA + `_http._tcp` on the selected interfaces |
| Per-app names | With `apps on`, every registered host of shape `<label>.local` is advertised (A/AAAA + `_https._tcp` port 443); registration adds it, PATCH host changes add/withdraw the diff, DELETE and heartbeat reap withdraw all of the app's names — each with TTL-0 goodbyes |
| `apps off` | Only the configured name is advertised; app registrations advertise nothing (and `skipped_hosts` does not count them — suppression is not skipping) |
| Skipped hosts | Non-`.local` hosts: never advertised, not counted. Multi-label `.local` hosts: not advertisable, counted in `skipped_hosts`, logged once per registration |
| Conflict | Library renames per RFC 6762; effective name logged at WARN and surfaced on `/1.0/mdns` and the status page |
| Asynchrony | No multicast I/O in any registry critical section — the reconcile goroutine converges add/withdraw after the mutation returns |
| Address change | Answers reflect live interface addresses per query; no proactive re-announcement in v1; peer staleness bounded by the 120s record TTL |

### Dashboard

| Row | Contract |
| --- | --- |
| Read-only | `GET`/`HEAD` on `/` and `/status.json` only; all else 404/405; zero mutating endpoints exist |
| Never a `/1.0` proxy | The status handler reads registry, data plane, cache, and hub state in-process; it never dials, embeds, or forwards to a control listener — no path from the LAN dashboard to control-plane authority, by construction |
| Redaction | No upstream socket path bytes, no `bridge_path`, anywhere in HTML or JSON |
| Self-contained | Zero external resources; renders on an offline LAN |
| Freshness | Page polls every 10s; `/status.json` is a point-in-time snapshot (counters monotonic, not mutually atomic — the `/1.0/cache` discipline) |

### Redirect + diagnostic

| Row | Contract |
| --- | --- |
| No canonical | No probe, no redirect — the dashboard, nothing else |
| Probe | Client-side `no-cors` fetch of the canonical origin, ~3s timeout; completion (any status) = reachable |
| Reachable | `location.replace(canonical)` |
| Unreachable | One-screen explanation with the router fix table; dashboard still rendered beneath |
| Loop guard | Same-origin as canonical → no probe, no redirect |
| Never server-side | The front door always answers 200 with the page; no 3xx is ever emitted — reachability is a client-side fact |

### Lifecycle

| Row | Contract |
| --- | --- |
| Pooled state | The advertiser (responder, registrations, reconcile goroutine) lives in the pooled process state (`caddy.UsagePool`), beside registry, data plane, and hubs |
| Config reload, mdns unchanged | **No LAN flap:** the new config's Provision reconciles against the live advertiser, finds no diff, and no goodbye or re-probe is ever multicast — the cache/hub reload discipline applied to the airwaves |
| Config reload, `name` changed | Old name withdraws (goodbye); new name probes and announces; per-app names untouched |
| Config reload, mdns removed | Real teardown: every record withdraws with goodbyes; the front-door listener closes |
| Config reload, front-door options changed | The listener follows Caddy's listener pooling (the control-listener model): unchanged address is reused, never rebound |
| Process exit | The pooled holder's Destruct removes every registration (explicit goodbyes), then stops the responder — the LAN sees a clean departure, not a timeout |
| Janus restart | The registry story: memory-only, the LAN forgets us on exit and learns us again when tenants re-register; the configured name returns as soon as the process starts |

### Security

| Row | Contract |
| --- | --- |
| Off by default | Enabling mdns is an explicit operator statement: *announce this machine and its app list to the local network.* The capability doc and `Caddyfile.example` say so in those words |
| What the LAN learns | The advertised `.local` names via multicast; and, from the dashboard, the full app list — names, all hosts (including non-`.local`), health, counters. Anyone who can reach `:80` on this machine can read it |
| No auth on the front door | v1 posture, stated plainly: the dashboard is exactly as public as the LAN it serves. Operators who need less than that do not enable mdns (or firewall `:80`) |
| Read-only, enforced | Zero mutating endpoints; no `/1.0` proxying or embedding (rows above) — possession of the dashboard confers observation, never control |
| Redaction | Socket paths and `bridge_path` never leave the process on this surface |
| Multicast trust | mDNS is unauthenticated by nature; any LAN peer can answer or squat names. Probing makes squatting visible (conflict rename, logged); it cannot make it impossible. LAN-hostile environments are outside this capability's threat model, and the doc says so rather than implying otherwise |

## The HTTPS-everywhere tier (configuration, not code)

Everything in this section is stock Caddy configuration plus two
build-time modules — no Janus code path. It is documented here because
it is the worked pattern the `canonical` option exists to hand off to,
and because its failure modes are the diagnostic page's subject
matter. The pattern, with Route 53 (any `caddy-dns` provider works the
same way):

1. **DNS, set once, static:** a wildcard CNAME `*.lan.<domain>` →
   `lan.<domain>`, and one A record `lan.<domain>` that the server
   keeps current itself.
2. **Self-updating A record:** the stock `caddy-dynamicdns` module
   with `ip_source interface` and the private-subnet ranges, so the
   record follows the machine's LAN address:

   ```caddyfile
   {
   	dynamic_dns {
   		provider route53
   		domains {
   			<domain> lan
   		}
   		ip_source interface en0
   		versions ipv4
   	}
   }
   ```

3. **Wildcard certificate, DNS-01:** the name never needs to be
   publicly reachable — ACME talks to the DNS API, not the host:

   ```caddyfile
   *.lan.<domain> {
   	tls {
   		dns route53
   	}
   	janus
   }
   ```

4. **Scoped IAM:** the credential on the box can touch exactly two
   records and nothing else —
   `route53:ChangeResourceRecordSetsNormalizedRecordNames` conditioned
   to `_acme-challenge.lan.<domain>` (TXT) and `lan.<domain>`
   (A/AAAA), on the one hosted zone.
5. **Build:** `xcaddy build --with github.com/shreeve/janus=.
   --with github.com/caddy-dns/route53
   --with github.com/mholt/caddy-dynamicdns --output ./bin/caddy`.
6. **The canonical dashboard, one line:** a site block for the
   canonical name reverse-proxying the front door —
   `janus.lan.<domain> { tls { dns route53 } reverse_proxy
   127.0.0.1:80 }` — so the redirect lands on this same status page
   over trusted HTTPS (the loop guard above exists for exactly this).

Result: every device on the LAN — no cert install, no profile, no
ritual — gets green-padlock HTTPS to `*.lan.<domain>` names that
resolve to a private address.

### Honest caveats

| Caveat | Reality | Remedy |
| --- | --- | --- |
| Offline LAN | No internet → no public DNS → `*.lan.<domain>` does not resolve | The `.local` front door still works — that is why mdns exists alongside this tier, not instead of it |
| DNS rebinding filters | Public names resolving to private addresses are silently dropped by default on Fritz!Box, pfSense, OPNsense, OpenWrt, and Pi-hole; the symptom is a timeout, not an error | One-line whitelist of `lan.<domain>` in the router's rebind-protection settings — the diagnostic page names the router families and the exact setting |
| Google WiFi | No rebind whitelist exists | The `.local` front door and `tls internal` are the honest fallbacks; the diagnostic page says so instead of pretending |
| No domain at all | Nothing to CNAME, nothing to DNS-01 | `tls internal` (Caddy's built-in CA) over `.local` or LAN names — with the per-device trust ritual stated honestly: export the root, install it, and on iOS additionally flip Certificate Trust in Settings, per device. Workable for one person's devices; miserable past that. The pattern above exists because this ritual does not scale |
| Split-horizon purists | The public zone carries a private RFC 1918 address | True and deliberate; it leaks the LAN's numbering plan and nothing else. Operators who object run the same pattern on an internal resolver |

## Registry, control-surface, and repo deltas

1. **The registry** gains one notification hook (the
   `purge`/`hubTeardown` pattern — wired once by the pooled state
   holder, invoked outside the lock on create, host-changing PATCH,
   DELETE, and TTL reap) and one in-process read-only accessor for
   per-app heartbeat age. The `/1.0/apps` JSON representation is
   unchanged.
2. **The data plane** gains a read-only per-path unhealthy-flag
   snapshot (atomic reads of existing state; no locks, no hot-path
   touch).
3. **The control surface** gains `GET {base}/1.0/mdns` on every
   control listener (existing Bearer behavior) — the acceptance
   oracle and the operator's view of advertiser state:

   ```json
   {
     "enabled": true,
     "name": "janus.local",
     "effective_name": "janus.local",
     "canonical": "https://janus.lan.ripdev.io",
     "interfaces": ["en0"],
     "front_door": "http://0.0.0.0:80/",
     "advertised": [
       { "name": "janus.local", "state": "announced" },
       { "name": "shop.local", "state": "announced", "app": "shop-x7k2p9" }
     ],
     "skipped_hosts": 0
   }
   ```

   With mdns disabled the endpoint answers `{ "enabled": false }` —
   present and honest, like `/1.0/cache` with no cache-enabled sites.
   The handler lands in `control_mdns.go` (the `control_hub.go`
   precedent); `GET /1.0`'s body gains an `"mdns"` boolean beside
   `"ping"`.
4. **Implementation shape** (owner's taste: few files, clean code):
   `mdns.go` (advertiser lifecycle, front door, status handler),
   `mdns_config.go` (parse, provision, validation), `mdns.html`
   (embedded page), `mdns_test.go`, `control_mdns.go` (hot surface).
   No new Caddy module: mdns is `App` config (an `Mdns` settings
   struct beside `Control`), parsed in `caddyfile.go`'s global block,
   rejected in `handler.go`'s site block.
5. **`test.sh`** gains the `mdns` group, appended after `tenant`
   (groups run in capability landing order); AGENTS.md's group list
   and capability table gain the matching entries; the root
   `Caddyfile` exercises the capability per the loop's step 4.
6. **The pool protocol and cache/hub contracts are untouched** — mdns
   observes; it participates in no data-plane decision.

## Non-goals (v1)

Each a decision, not an omission:

- **No hub-powered live dashboard** — the page polls; it opens no
  WebSocket. A system channel carrying registry events is named as
  future work, not smuggled in here.
- **No DNS server** — Janus answers multicast queries for its own
  names; it resolves nothing, forwards nothing, and serves no unicast
  DNS.
- **No hosted certificate service** — the HTTPS tier is the operator's
  DNS provider and ACME; Janus mints nothing beyond what stock Caddy
  already does.
- **No advertising of non-`.local` names** — multicast is for the
  mDNS domain; public names belong to real DNS.
- **No write path on the dashboard** — no buttons, no forms, no
  actions, ever on this surface; steering is the control plane's job.
- **No auth on the front door** — v1's posture is LAN-public or
  disabled; a token-gated dashboard is a different product.
- **No proactive re-announce on address change** — query-time
  freshness plus record TTL (mechanism section); an interface watcher
  is future work.
- **No `listen` knob for the front door** — `:80` fixed; flagged in
  the ledger for review.
- **No TXT-record payloads** — the service advertisements carry no
  metadata; the dashboard is the metadata.

**Future work** (explicitly deferred, in the sibling docs' sense):
system-channel registry events for a live dashboard; an interface
watcher for proactive re-announcement.

## Acceptance sketch (`test.sh` mdns group)

Multicast itself is not CI-assertable — `test.sh` runs on machines
and networks where multicast may be filtered, absent, or shared with
real Bonjour traffic, so no case below sends or receives a multicast
packet. The group asserts the **advertiser's state** (via
`/1.0/mdns`) and the **front door's behavior** (via direct HTTP to the
listener); the multicast wire behavior is the library's Bonjour
Conformance-tested territory, verified manually per release, stated
here so nobody later "fixes" the suite by adding a flaky multicast
listener.

| Case | Assert |
| --- | --- |
| **Disabled default** | Config without `mdns`: `/1.0/mdns` → `{"enabled": false}`; nothing listens on the front-door port |
| **Enabled minimal** | `mdns` present: `/1.0/mdns` shows `janus.local` advertised; `GET /` on the front door → 200 HTML; `GET /status.json` → 200 JSON with the expected top-level keys |
| **Registration advertises** | Register an app with `hosts: ["shop.local", "shop.ripdev.io"]` → `/1.0/mdns` advertised gains `shop.local` (with the app id) and never `shop.ripdev.io` |
| **Multi-label skipped** | Register `hosts: ["a.b.local"]` → not advertised; `skipped_hosts` = 1 |
| **DELETE withdraws** | Delete the app → `shop.local` gone from advertised |
| **Reap withdraws** | Short-TTL harness (`JANUS_HEARTBEAT_TTL`); let the heartbeat lapse → advertised set loses the app's names |
| **PATCH reconciles** | PATCH hosts swapping `shop.local` for `store.local` → advertised set reflects exactly the diff |
| **`apps off`** | Same registrations advertise nothing; only the configured name is present |
| **Status page truth** | Register an app, PUT upstreams, heartbeat: `/status.json` shows the app with correct host list, upstream total/healthy, and a small `heartbeat_age_ms` |
| **Redaction** | Register with a distinctive socket path (`/tmp/SENTINEL-xyz.sock`): those bytes appear nowhere in `/status.json` or `GET /` |
| **Read-only** | `POST /`, `POST /status.json`, `PUT /status.json` → 405; `GET /anything-else` → 404 |
| **Canonical present** | Config with `canonical`: `/status.json` carries it; the served HTML contains the probe script and the canonical origin |
| **Reload no-flap** | Reload an identical config → `/1.0/mdns` advertised set identical, and the advertiser reports zero withdraw/re-announce cycles across the reload |
| **Reload teardown** | Reload with `mdns` removed → front-door port closed; `/1.0/mdns` → `{"enabled": false}` |
| **Parse rejections** | Each parse hard-error line above fails `caddy adapt` with a precise error (site-level `mdns`, `mdns on`, bad `name` forms, bad `canonical` forms, `interface` with no args, `apps maybe`, duplicates, nested blocks) |

### Go-test surface

`go test ./...` units, one per contract row:

- **Parse table** — every legal line accepted; every parse hard-error
  row rejected with its message (the cache/hub config-test pattern);
  name lowercasing; site-level rejection.
- **Name validation edges** — 63-byte label at the boundary, 64
  rejected; hyphen placement; `janus.local.` trailing dot; unicode
  rejected.
- **Canonical validation edges** — port accepted; path/query/fragment
  /userinfo rejected; `http://` rejected; `.local` target rejected.
- **Desired-set computation** — configured name ∪ advertisable hosts;
  single-label-`.local` classifier (table-driven: `x.local` yes,
  `x.y.local` no, `x.ripdev.io` no, `local` no); `apps off`
  suppression; dedup when an app host equals the configured name.
- **Reconcile state transitions** — against a fake responder: create
  → add; PATCH diff → add + remove; DELETE/reap → remove; no-diff
  reload → zero responder calls (the no-flap pin); mutation-side hook
  never blocks on reconcile (asynchrony pin, `-race`).
- **Snapshot redaction** — an app with socket paths and a bridge_path
  set: serialized `/status.json` bytes contain neither.
- **Heartbeat age accessor** — fake clock; age advances; re-stamp
  resets; accessor takes only a read lock.
- **Front-door routing** — `/` and `/status.json` GET/HEAD 200;
  method table 405; path table 404.
- **Provision/Start errors** — missing pinned interface; HTTP-app
  listener collision (table of colliding and non-colliding configs).

## Measurement

No performance ledger entry: this capability makes no performance
claim and never touches the data plane's request path (rule 8 is
satisfied by the absence of claims, not by a benchmark of a page
served at human refresh rates). The one measurable pin is structural
— advertising work happens on the reconcile goroutine, off every
registration/reap critical section — and it is pinned by the
asynchrony unit test, not by a bench.

## Open questions for adversarial review

Decisions this revision made that the settled design did not specify,
plus one correction of it. Numbered for the review's convenience;
resolutions fold into revision 2.

1. **IP-change behavior corrects the settled design.** "IP change
   re-announces" is not a service `brutella/dnssd` provides (hot-plug
   is open in its own TODO). Revision 1 pins query-time freshness with
   TTL-bounded peer staleness (≤ 120s) and defers a proactive
   interface watcher. Is that window acceptable for the DHCP-renewal
   case, or must v1 carry its own watcher (poll interfaces, withdraw +
   re-add on change — at the cost of a visible goodbye/probe flap on
   every change)?
2. **Advertising mechanism.** No bare-hostname API exists in the
   library; A/AAAA records ride on service registrations (front door:
   `_http._tcp` 80; app hosts: `_https._tcp` 443). Are those
   type/port choices right for app hosts admitted only on the
   on-demand `:8443` site?
3. **Front-door listen is fixed at `http://0.0.0.0:80/`** with hard
   Start errors on bind failure or HTTP-app collision, and no `listen`
   knob. The portless URL is the product; but review should weigh the
   operator who cannot have `:80` (and note `test.sh` must bind it on
   CI machines).
4. **Grammar has no `on|off` argument** (presence enables) — unlike
   cache/hub, like nothing else in the block. Consistency versus
   cascade-lessness: is `mdns off` worth accepting as a no-op-with-
   documentation, or is rejecting it (as revision 1 does) the right
   loudness?
5. **Conflict rename is tolerated loudly** (WARN + surfaced on
   `/1.0/mdns` and the page) rather than treated as an error state.
   The alternative — refusing to advertise under a renamed identity —
   trades availability for strictness; review should confirm the
   tolerance is right for a mechanism whose RFC mandates renaming.
6. **Multi-label `.local` hosts are skipped and counted, not
   rejected at registration.** Rejecting would couple hot registration
   to cold mdns state (a registration legal today becomes 400 when the
   operator enables mdns). Confirm the skip-and-count posture.
7. **The front door checks no Host header** and serves the dashboard
   to any name or bare IP that reaches the listener. Deliberate (the
   diagnostic story wants IP-literal access); review the
   information-exposure trade against a strict-Host 404.
8. **No auth on the dashboard, and it lists *all* apps and hosts** —
   including non-`.local` public hostnames — to any LAN client.
   Sharper than the multicast announcement alone (which only carries
   `.local` names). Acceptable v1 posture, or does the app list need
   an opt-out (`apps off` currently governs advertising only, not the
   page)?
9. **`heartbeat_age_ms` on `/status.json` and `/1.0/mdns`** is the
   explicit registry surface addition (accessor stays in-process;
   `/1.0/apps` unchanged). Confirm age-not-timestamp is the right
   shape and that `/1.0/apps` should stay unchanged.
10. **`interface` accepts multiple names** (the library takes a list);
    a missing pinned interface is a hard Start error even though
    interfaces come and go on laptops. Confirm both.
11. **Auto-select interface rule** — up + multicast + non-loopback +
    has a usable unicast address, with `127/8`, `::1/128`,
    `169.254/16` blocked, IPv6 link-local advertised. The library's
    own default does not exclude loopback; revision 1 excludes it
    explicitly. Confirm the block list.
12. **`canonical` validation strictness** — origin-only (no path),
    never `.local`. Confirm; and confirm the client-side-only redirect
    (no server 3xx ever) survives review.
13. **`/1.0/mdns` as the acceptance oracle** — a new control surface
    (in `control_mdns.go`) that exists primarily because multicast is
    not CI-assertable, secondarily for operators. Confirm the surface
    and its disabled-state shape.
14. **Fixed 10s poll, fixed ~3s probe timeout, CDN-free page** — all
    pinned without knobs. Confirm zero-external-resources as a
    contract row (the offline-LAN claim depends on it).
