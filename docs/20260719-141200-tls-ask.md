# TLS ask: on-demand certificate allowlist

Hot `/1.0` surface riding the existing **control** capability — not a new
cold capability. Caddy's On-Demand TLS asks Janus before minting a
certificate; Janus answers from the apps registry.

| | |
| --- | --- |
| **Phase** | 6 (build SPEC) |
| **Surface** | Hot `/1.0` (control listeners) + stock Caddyfile TLS grammar |
| **Cascades** | n/a — not a cold capability |
| **Module** | `janus` app (`control_api.go`); registry (`apps.go`) |
| **Endpoint** | `GET {base}/1.0/tls/ask?domain=<name>` |
| **Status** | Served on every control listener |

## Why it exists

Cold Caddyfile config owns the TLS machinery: listeners, issuers, ACME,
and which sites opt into on-demand minting. What it cannot know is
*which names are legitimate right now* — that is registry state. The ask
endpoint is the one-question bridge: **may we mint for this name?** The
hot registry answers; it never touches listeners or issuers.

## Contract

| Request | Response |
| --- | --- |
| `domain` is a host claimed by a registered app | **200** `{"domain":"…","app":"<id>"}` |
| `domain` is not a registered host | **404** `{"error":"domain \"…\" is not a host of any registered app"}` |
| `domain` missing or empty | **400** |
| Any method but `GET` | **405** |

Caddy allows minting on **200** and denies on any other status. Janus
answers **404** for unknown names — the same status the data plane gives
an unknown host, and the same "there is no such resource" meaning as an
unknown app id. (403 would imply an authenticated caller was refused;
nothing here is about the caller.)

### Matching

- **Exact match only.** The query is normalized the way the data plane
  normalizes `Host` — lowercased, trailing dot stripped, any `:port`
  dropped — then looked up in the registry's host index. Registered
  hosts are already stored lowercase (Phase 3).
- **No wildcards.** Phase 3 host validation rejects `*` in any label, so
  a wildcard host can never enter the registry, and a wildcard ask can
  never match. On-demand names are concrete SNI values anyway; a tenant
  that wants ten names registers ten hosts.

### Lifecycle

Allowance is registry membership — nothing separate to manage:

| Transition | Ask answer |
| --- | --- |
| `POST /1.0/apps` (host claimed) | 200 |
| `DELETE /1.0/apps/{id}` | 404 |
| Heartbeat TTL reap | 404 |
| Alive but not routable (`upstreams []`, fresh heartbeats) | **200** — heartbeat ≠ readiness; a reload never breaks TLS |

### Auth

The ask rides the control listeners with the same Bearer behavior as
every `/1.0` route. Caddy's `ask` request carries no Authorization
header, so point it at a listener without `token:…` (the default
`local` listener works). Pointing it at a tokened listener denies every
mint with 401 — loud, visible in both logs, nothing minted silently.

## Cold config

Normal Caddy grammar only — no new directives. The global option names
the ask URL; each site opts in per stock `tls` syntax:

```caddyfile
{
	on_demand_tls {
		ask http://127.0.0.1:7600/1.0/tls/ask
	}

	janus {
		control local
	}
}

# Public: ACME issuers mint on demand for registered hosts
https:// {
	tls {
		on_demand
	}
	janus
}
```

The repo-root `Caddyfile` demonstrates the path with the internal CA on
`:8443` (no ACME dependency):

```caddyfile
https://:8443 {
	tls {
		on_demand
		issuer internal
	}
	janus
}
```

One sharp edge to know: Caddy consults its certificate cache **before**
on-demand. A name already covered by a cold-loaded certificate (e.g. the
committed `*.ripdev.io` wildcard) is served that certificate and the ask
never fires. The ask path is exercised by names the cold config does not
cover — which is exactly its production role.

## Verify

```bash
go test ./...
./test.sh   # tls group

# By hand, against the running root Caddyfile:
curl -s -o /dev/null -w '%{http_code}\n' \
  'http://127.0.0.1:7600/1.0/tls/ask?domain=demo.janus.test'   # → 404

curl -s -X POST -H 'Content-Type: application/json' \
  --data '{"name":"demo","hosts":["demo.janus.test"]}' \
  http://127.0.0.1:7600/1.0/apps                               # → 201

curl -s -o /dev/null -w '%{http_code}\n' \
  'http://127.0.0.1:7600/1.0/tls/ask?domain=demo.janus.test'   # → 200

# Handshake: cert minted on demand from the internal CA
echo | openssl s_client -connect 127.0.0.1:8443 \
  -servername demo.janus.test 2>/dev/null |
  openssl x509 -noout -issuer -ext subjectAltName
# → issuer=…Caddy Local Authority…, DNS:demo.janus.test
```

## Hard errors

- Missing or empty `domain` → 400
- Non-GET method → 405
- Wildcard hosts rejected at registration (Phase 3), so never allowed here

## Non-goals

- Per-app issuer or TLS policy (cold config owns the machinery)
- Wildcard allowances
- Persisting allowances (registry is memory-only; tenants re-register)
- Rate limiting the ask (Caddy already rate-limits on-demand minting)
