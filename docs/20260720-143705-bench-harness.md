# Bench harness — reconstructing the measurement rig

How to stand up the full-stack bench rig (oha → Janus/TLS → UDS → Bun
worker) and run legs that are comparable to the recorded results. The
canonical numbers live in
[20260719-165500-rip-server-performance.md](20260719-165500-rip-server-performance.md)
("Canonical cold-machine baseline"); this page is the *how*, kept so a
future session does not have to reverse-engineer the rig from raw
provenance files again (this page exists because one did).

The harness runs from a scratch directory (`/tmp/janus-bench` in the
scripts below); only docs and raw results land in the repo. Raw result
files are `docs/YYYYMMDD-HHMMSS-bench-raw-*.txt` — flat under `docs/`
(a `docs/bench/` subfolder is a pending policy call; do not create it
unilaterally).

## The tenant

One app, two hosts through identical TLS and identical workers, so
cache off/on legs differ by nothing but the site's `cache` setting in
the root `Caddyfile`:

- `bench.ripdev.io` — inherits global cache **on** (ttl 1s)
- `api.ripdev.io` — site block sets `cache off`

`app.rip` (scratch dir):

```coffee
import { get, start } from '@rip-lang/server'

get '/' -> { ok: true }

get '/io' ->
  sleep! 5
  { ok: true, io: true }

start()
```

`/` is the ping-class route (Janus-bound); `/io` is the 5ms
capacity-bound route (clean 200s/s ≈ w×c/5ms at c-undersized configs,
503s shed above it).

The scratch dir needs `@rip-lang/server` resolvable — the manager's
`Bun.build` artifact step resolves imports from the app's directory:

```bash
mkdir -p /tmp/janus-bench/node_modules/@rip-lang
ln -sfn ~/Data/Code/rip/packages/server   /tmp/janus-bench/node_modules/@rip-lang/server
ln -sfn ~/Data/Code/rip/packages/validate /tmp/janus-bench/node_modules/@rip-lang/validate
```

## Standing up the stack

```bash
cd ~/Data/Code/janus
export PATH="$(go env GOPATH)/bin:$PATH"
xcaddy build --with github.com/shreeve/janus=. --output ./bin/caddy
ulimit -n 65536   # or higher; on every shell: janus, manager, bench
./bin/caddy run   # root Caddyfile: control on 127.0.0.1:7600
```

Manager, from the scratch dir — invoke bun DIRECTLY, not the
`rip-server` bin:

```bash
RIP=~/Data/Code/rip
RIP_ENV=production bun \
  --preload=$RIP/src/loader.js $RIP/src/run.js \
  $RIP/node_modules/.bin/rip-server app.rip \
  --name bench --host bench.ripdev.io --host api.ripdev.io \
  -w 2 -c 1 --control http://127.0.0.1:7600
```

**Why direct bun:** the `rip-server` bin runs under the `rip` CLI
wrapper, which spawns the real manager as a child — killing the
wrapper's pid orphans a live manager + workers that keep the host
claim. With direct invocation, `$!` IS the manager; SIGTERM drains and
deregisters cleanly. If an orphan does hold the claim, the next
register 409-retries for `RIP_REGISTER_409_MS` (20s default), riding
out heartbeat reap — a sweep that restarts the manager per config can
just poll readiness with `curl -sf https://api.ripdev.io/`.

## Running legs

oha 1.14.0 sharp edges, all learned the hard way:

- **Unset `NO_COLOR`** (`env -u NO_COLOR oha …`). Agent/CI shells set
  `NO_COLOR=1`; oha treats it as `--no-color 1` and dies with
  `invalid value '1' for '--no-color'`.
- Machine-readable output is `--output-format json` (there is no
  `-j`); parse `summary.requestsPerSec`, `statusCodeDistribution`,
  `latencyPercentiles.p50/.p99` (seconds — ×1000 for ms).
- `--unix-socket <path>` gives the direct-to-worker attribution row
  (`http://localhost/` as the URL; non-HTTPS only). Worker socket
  paths come from `GET /1.0/apps` → `upstreams[].path`.
- Discipline per the perf doc: 15s legs, one discarded 5s warmup after
  every config change, `--no-tui`, keep-alive default.

Report clean-200 throughput (`200s/s`) separately from raw RPS on
capacity-bound routes — 503 sheds are fast and inflate RPS.

Cache truth comes from `GET /1.0/cache` counter deltas around each
leg (`misses`/`coalesced`/`hits`/`bypass`/`waiter_overflow`), which is
how "worker saw ~1 req/s" claims are grounded.

## The canonical sweep

The baseline runner that produced
[20260720-090645-bench-raw-canonical-baseline.txt](20260720-090645-bench-raw-canonical-baseline.txt),
in full — four sections in fixed order, manager restarted per config,
everything `tee`'d to the raw file:

- **A) w sweep** — ping-class, cache off, c:1; w ∈ {2,4,8,16,32};
  legs at conc=w and conc:64; worker-RSS snapshot per w.
- **B) c sweep** — `/io`, w:8, cache off; c ∈ {1,4,8,16} at conc:64,
  plus conc:128 at c:16 (capacity = w×c must reach conc for clean
  legs).
- **C) cache off/on** — w:2 c:1, conc:64, interleaved
  off/on/on/off pairs for `/io` then `/`, cache-counter deltas on
  every on-leg.
- **D) attribution** — direct worker UDS at conc 1 and 2, then Janus
  TLS at conc:1.

Log `uptime` load averages at start and between sections; on a
cold-booted, otherwise-idle machine identical-config legs agree
within ±3% (warm, loaded sessions drifted ±10–24% — on those, only
interleaved ratios are trustworthy).

## Leak probe (RSS questions)

The test for "leak or steady state" is slope vs plateau against
CUMULATIVE requests: hammer ONE worker direct-UDS in successive 15s
batches, `ps -o rss= -p <worker pid>` after each (worker pid via
`lsof -t <socket>`). Measured 2026-07-20: 31.1MB at publish, 77.2MB
after the first ~1.7M requests, 77.5MB after 13.7M — all growth in
batch one, +0.3MB over the next 12M. JSC sizes its heap to allocation
rate and keeps freed pages resident at the high-water mark, so
steady-state RSS tracks request RATE, not request COUNT — which is
also why at-boot RSS (33–40MB), w:32 workers (52–54MB), and
max-rate w:2 workers (69–77MB) are all the same healthy heap at
different loads. A slope that keeps climbing across batches is the
defect signature; a plateau is not a finding.

## Session hygiene

- The manager must be killed by ITS pid (see direct-bun note above);
  verify teardown with `pgrep -fl 'caddy run|rip-server|worker.js'`.
- Managers started from throwaway automation shells die with the
  shell — run caddy and the manager in their own long-lived shells
  (or let the sweep script own the manager lifecycle).
- Rebuild `bin/caddy` after any Go change and note the janus commit
  in the raw file's header; the manager runs whatever the rip
  checkout's main is — note that too when it matters.
