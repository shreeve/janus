# Capability: hub (Phase 7 — per-app WebSocket fan-out)

> **The written contract for Phase 7.** This document is what the
> implementation and acceptance tests build against. Protocol ancestry is Bam
> (`../bam/src/bam.cr`, reference implementation; `ideas.cr`, sketch;
> `bampub.cr`, publish CLI) — behavior pins port from **`bam.cr`**
> except where a divergence is stated explicitly and framed honestly.
> Owner decisions appear as DECIDED and are not reopened here.

Cold Caddyfile capability plus hot-plane state. When enabled for a
site, Janus accepts WebSocket connections on the hub path, terminates
them at the edge, and fans JSON directive frames out to channels and
connections — per app, in memory, with the tenant observing and
steering through an HTTP bridge and a control-plane publish endpoint.

| | |
| --- | --- |
| **Order** | **4** (after **ping**, **control**, **cache**; sits on the Phase 4 data plane and the Phase 3 registry) |
| **Surface** | Cold config (Caddyfile) for capability + admission; hot `/1.0` for `bridge_path`, publish, and counters |
| **Cascades** | Yes — global default → site override |
| **Built-in default** | off (when unset at every level) |
| **Module** | Site handler (data plane), upgrade interception before cache and upstream selection |
| **Status** | **Shipped** — implemented as written |

## Why it exists

Workers are disposable by contract: a reload scraps the pool, socket
paths are never reused, and the pool protocol's admission cut kills
old workers in seconds. A WebSocket held by a worker dies with it.
The hub inverts the ownership: **connections terminate at Janus** —
the edge never restarts during app reloads — and the tenant
participates over plain HTTP, which the data plane already routes,
retries, and wakes (doorbell) correctly. Clients keep their sockets
across every reload; workers keep their disposability.

The protocol is Bam's: a ~340-line Crystal proxy that defines `@`
(targets), `+`/`-` (join/leave), opaque event bundles, an HTTP bridge
carrying `Sec-WebSocket-Frame: open|text|close`, and a `/publish`
endpoint. Hub is a Go reimplementation of that protocol on the Janus
chassis: per-app namespaces, registry-backed admission, the existing
control plane for publish, the existing data plane for the bridge.

## The three planes

Directive frames — the JSON grammar below — arrive from three places
and are executed by **one** executor with per-plane policy:

| Plane | Carrier | Trust | Context |
| --- | --- | --- | --- |
| **Client** | WebSocket text frame | Untrusted | The sending connection: `@` defaults to it, `<` is stamped from it, `!`-suffix exclusion keys off it |
| **Bridge response** | Body of the tenant's 2xx answer to a bridge POST | Trusted (the tenant) | The originating connection of the bridged frame (for `open`/`text`/`close` alike) |
| **Publish** | Body of `POST /1.0/apps/{id}/hub/publish` | Trusted (control plane) | No originating connection |

One grammar, three sources. Rows in the sigil table state what each
plane may say; anything a plane may not say is rejected loudly, never
stripped.

Client frames execute **at the edge**: Janus parses, validates, and
applies them (membership, fan-out, liveness) without waiting on the
tenant. Every client frame is also forwarded to the tenant's bridge
for observation and reaction (bridge contract below) — but delivery
never gates on the bridge. This is the property that makes the hub
survive reloads: fan-out keeps working while the pool reboots.

## Wire grammar

**DECIDED: the wire model is JSON objects (or lists of objects) whose
keys are sigils or event names** — direct Bam ancestry. A frame is:

- one JSON object, or
- a JSON array of objects; every canonical stage preserves list order.

Anything else — a string, a number, a bare array of non-objects,
binary data — is malformed (hard errors below). Every key in an
object is exactly one of: a **sigil** (the table below), or an
**event name** (grammar below). There is no third kind; a key that is
neither is malformed.

**Canonical execution order.** Every plane uses this order, and all
later sections refer back to it rather than redefining it:

1. Parse and validate the **whole frame** — every object, key, value
   shape, plane policy, collision, and resource limit. Any violation
   rejects the whole frame before any effect.
2. Apply `+`/`-` membership mutations in object-list order. Client
   mutations are sender-scoped. Trusted-plane mutation-subject
   selection expands its `@` only for the mutation in this stage;
   that subject selection is distinct from event-delivery resolution.
3. Resolve each object's `@` delivery targets against
   **post-mutation** membership.
4. Deliver event bundles, then answer `?`.

Key order inside a JSON object never changes this order. Objects in a
list preserve list order within each stage.

Example client frames:

```json
{"+": ["/lobby", "/game/42"]}
{"@": ["/game/42"], "move": {"x": 3, "y": 5}}
{"@": ["/game/42"], "move!": {"x": 3, "y": 5}}
{"@": ["k7f2m9x0q4w1z8p3"], "whisper": {"text": "hi"}}
{"?": "t1721512345"}
[{"-": ["/lobby"]}, {"@": ["/game/42"], "left": {"who": "…"}}]
```

Example delivered frame (recipient's view — `<` stamped by Janus,
`!` suffix stripped, sigil keys gone):

```json
{"<": ["k7f2m9x0q4w1z8p3"], "move": {"x": 3, "y": 5}}
```

### The sigil table

A sigil is a single-character key. Every sigil's disposition is
decided; `>` is reserved by decision, unused in v1.

| Sigil | Meaning | Client frame | Bridge response | Publish | Status |
| --- | --- | --- | --- | --- | --- |
| `@` | Event-delivery targets: channels and connection ids, one list | Legal; absent → sender; present-empty rejects | Legal; absent → originating connection; present-empty rejects | Legal; absent or present-empty → **400** | DECIDED |
| `+` | Join membership subjects to named channels | Legal only for the sender; `@` never changes its subject | Legal; `@` selects subjects and may expand channels | Legal; `@` selects subjects and may expand channels | DECIDED |
| `-` | Leave membership subjects from named channels | Legal only for the sender; `@` never changes its subject | Legal; `@` selects subjects | Legal; `@` selects subjects | DECIDED |
| `<` | Sender provenance list: an array of relay identities | **Rejected loudly** (Janus stamps `[connection-id]`) | Legal array; a relay may append | Legal array (`bampub.cr` ancestry) | DECIDED |
| `>` | — | **Rejected loudly** | **Rejected loudly** | **400** | **RESERVED, unused in v1** |
| `?` | App-level ping; Janus answers `!` at the edge, never touching the worker | Legal | **Rejected loudly** | **400** | DECIDED |
| `!` (exact key) | App-level pong; Janus → client only | **Rejected loudly** | **Rejected loudly** | **400** | DECIDED |
| `*` | Tenant kick-with-reason, wire delivery direction only | **Rejected loudly** | Legal | Legal | DECIDED |

Notes pinned by the decisions:

- **`@` entries self-discriminate:** a string starting with `/` is a
  channel (expands to its member set); any other string is a direct
  connection id. One list, no separate sigils per target kind.
  Type-based discrimination (ideas.cr's number = user) is
  **REJECTED**: it is serializer-fragile and silently misdelivers a
  stringified numeric id — the protocol equivalent of a silent
  miscompile.
- **Client `@` governs event delivery only.** Client `+`/`-` always
  mutate the sending connection. If a client object combines `+` or
  `-` with an `@` that resolves to anything other than exactly the
  sender, the frame rejects loudly. Cross-connection mutation and
  channel expansion as membership subjects exist only on the trusted
  bridge-response and publish planes.
- **`>` is reserved because `@` already answers "to whom."** Two
  spellings for targeting would violate one-way-to-do-each-thing.
  Reserved means rejected today: deployed traffic cannot use `>` as
  an event key, so a future meaning cannot collide.
- **`<` is an array-valued provenance list.** `bampub.cr`'s wire
  bytes use `"<":[sender]`. Janus stamps one client origin as
  `[connection-id]`; each trusted relay hop may append its identity.
  `ideas.cr`'s `>` sender sketch does not define this contract.
- **`+`/`-` values must be channels** (`/`-prefixed). "Join" means
  subscribe-to-channel; a connection id in a `+`/`-` list is
  malformed. (bam.cr's untyped MEMBERS map accepts client-to-client
  links; that tolerance does not port.) Bookkeeping is bidirectional
  as in bam.cr — connection→channels and channel→connections mirror
  each other — and **a channel whose member set empties is deleted**.
- Values of `@`, `+`, `-`, and `<` are JSON arrays of strings. Any
  other value shape is malformed. A present `@` must contain at least
  one entry on every plane.

### Names: channels, connection ids, event names

**Channels** are `/`-prefixed strings, nestable as names
(`/friends/high-school/math-team`):

- Grammar: `^(/[A-Za-z0-9._~-]+)+$` — one or more non-empty segments,
  each of URL-safe characters; no empty segments (`//`), no trailing
  `/`, no whitespace, no control characters.
- Length: ≤ 128 bytes total.
- **Hierarchy is NAMING-ONLY in v1 (DECIDED):** targeting `/friends`
  does **not** fan out to `/friends/high-school` members. A channel
  name is an opaque key; the `/` separators are for humans and for a
  future subtree decision (non-goals). Implicit subtree delivery and
  wildcards are explicit non-goals, default off.
- A channel exists exactly while its member set is non-empty. There
  is no create/destroy verb; the first join creates, the last leave
  (or disconnect) deletes.

**Connection ids** are minted by Janus, never chosen by a client:
16 characters of `[a-z0-9]` from `crypto/rand` (~82 bits — the id
doubles as an unguessable address; bam.cr's 5-hex-char id carries
`FIX` in its own source and does not port). On the wire a direct
target is any non-`/` string; a target that matches no live
connection id is skipped at delivery (churn makes stale ids normal)
and counted, never an error.

**Event names** are every key that is not a sigil:

- Grammar: `^[A-Za-z_][A-Za-z0-9._-]{0,63}!?$` — starts with a letter
  or underscore, so no event name can ever collide with a current or
  future sigil; ≤ 64 bytes before the optional `!` suffix.
- The trailing `!` is the include-sender delivery directive (next
  section), part of the wire spelling, never part of the name.
- A name may have at most one trailing `!`; `chat!!` is illegal.
- A frame carrying both bare and suffixed forms of one event name,
  including across different objects in a list, rejects as a whole
  before any effect. Stripping both to one delivered key would make
  the frame's meaning representation-dependent.
- An event's value is arbitrary JSON and is **never interpreted** —
  Janus delivers event bundles byte-preserved. This is bam.cr's
  pass-through model, kept: the hub routes; it does not understand
  payloads.
- One object may carry several event keys; they deliver together in
  one bundle (one WebSocket message per recipient), as in bam.cr.

### The `!` suffix (include-sender delivery)

**DECIDED — in for v1, with four pinned rules.** A bare event name
delivers to the resolved target set **excluding the sender**; the
`!` suffix delivers to the full set, sender included.

1. **Exclusion keys off the ORIGINATING CONNECTION only, never the
   user.** A sender's other open tab is a different connection and
   still receives bare-name deliveries. There is no user identity at
   this layer to key off.
2. **The suffix is a wire directive, STRIPPED on delivery.**
   Recipients always see `chat`, never `chat!`. A recipient cannot
   tell which spelling the sender used.
3. **Server-originated publishes have no originating connection, so
   bare and `!` spellings deliver identically there. Exclusion never
   keys off `<` attribution** — a bridge response or publish that
   carries `<` naming a live connection does not thereby exclude it.
   (Bridge responses *do* have an originating connection — the
   client whose frame they answer — so exclusion applies to them
   exactly as to client frames.)
4. **Exact-key `!` is pong.** Whole-key match versus a nonempty-name
   suffix — `!` alone is the liveness sigil, `chat!` is an event
   spelling; there is no ambiguity, and the grammar states it: the
   event-name regex requires at least one name character before the
   optional `!`.

A `!`-suffixed event whose only resolved recipient is the originating
connection produces one loopback delivery. A bare event whose only
resolved recipient is the originating connection produces no delivery;
that empty sender-excluded variant is legal and counted as zero.

An object with routing or provenance sigils but no event, membership
mutation, kick, or `?` is a no-op frame and rejects loudly. A frame
containing only `<` on a trusted plane therefore rejects rather than
pretending to do work.

**Honest framing:** this contract deliberately differs from bam.cr,
which has no sender-exclusion concept and delivers to the full target
set always. This contract adopts the convention from
ideas.cr: bare name excludes the sender by default, `!` includes
them. Consequence worth stating: a client frame with no `@` targets
the sending connection itself, so a bare-name event there delivers to
nobody, while `{"echo!": {...}}` delivers back to the sender — the
protocol's built-in loopback probe. Both are well-defined and legal.

### `<` attribution and `?`/`!` liveness

**`<` is EDGE-STAMPED (DECIDED).** On every client-originated
delivery, Janus sets `<` in the delivered bundle to
`[sending-connection-id]`. The array is provenance: each trusted
relay hop may append its identity while preserving earlier entries.
A client-SUPPLIED `<` is **rejected
loudly** (hard errors below): Janus already knows who the socket is,
so a client spelling `<` is either confused or spoofing, and both
reject. On the trusted planes (bridge response, publish) the sender
may supply an array-valued `<` — `bampub.cr` emits `"<":[sender]` —
and Janus passes it through unmodified; an absent `<` on a
trusted-plane delivery stays absent.

**`?` / `!` is app-level liveness (DECIDED).** Janus answers a client
frame's `?` with `{"!": <the ? value echoed verbatim>}` directly at
the edge, never touching the worker. Rationale: browser JavaScript
cannot send WebSocket-protocol-level pings, so in-band liveness is
the only option for browser clients; the echoed value lets a client
correlate (send a timestamp, measure round trip). `?` may ride in an
object with other directives; the pong is sent after the frame
executes. The `?` value must be a JSON string ≤ 128 bytes; anything
else is malformed.

**SEPARATELY, Janus keeps WebSocket-protocol-level ping/pong toward
clients** on a 30-second cadence (bam.cr's PING interval, kept) for
dead-peer detection: a connection that has not answered the previous
protocol ping by the time the next one is due is closed and cleaned
up (worst case ~60s to detect a silent peer). On ping timeout the
connection is closed via the internal close-with-reason mechanism
with Close 1001 `ping timeout`. Protocol-level pings
from clients (non-browser clients can send them) are answered by the
WebSocket layer as normal and are invisible to the hub grammar.

### Size and shape limits

Checked during validation, before anything executes. Every limit
rejects loudly; none clips silently.

| Limit | Default | On violation |
| --- | --- | --- |
| Frame size (one WebSocket message) | 64 KiB (`max_frame`, cold) | Close 1009 ("message too big") |
| Objects per list frame | 16 | Malformed → close 1008 |
| Targets per `@` list | 64 | Malformed → close 1008 |
| Channels per `+` / `-` list | 64 | Malformed → close 1008 |
| Event keys per object | 16 | Malformed → close 1008 |
| Channel name length / grammar | 128 bytes, regex above | Malformed → close 1008 |
| Event name length / grammar | 64 bytes + optional `!`, regex above | Malformed → close 1008 |
| `?` value | JSON string ≤ 128 bytes | Malformed → close 1008 |
| `*` value | JSON string ≤ 128 bytes | Client: always close 1008; trusted plane: malformed rejection |
| Channels per connection | 128 (`max_channels`, cold) | Whole frame rejects before any effect: close 1008 |
| Connections per app | 4096 (`max_conns`, cold) | Handshake rejected **503** + `Retry-After` before upgrade |

Binary WebSocket frames are rejected: close 1003 ("unsupported
data"). bam.cr logs and ignores binary; ignoring is
drift-tolerance, so v1 closes instead (non-goals: binary framing).
Text frames must be valid UTF-8 JSON — the JSON parser is the
arbiter; anything it rejects is malformed. The parser also rejects
duplicate object keys at any nesting level instead of accepting a
library-dependent first-wins or last-wins interpretation.

Trusted planes get the same shape limits with plane-appropriate
rejection: a bridge response that violates them is a **tenant
garbage** event (bridge failure modes below — directives dropped,
counted, logged, client unaffected); a publish body that violates
them is a **400** naming the offending object index, key, and rule.
Trusted planes are exempt from `max_frame` (they are bounded by the
control plane's existing 1 MiB body cap and by the data plane's
response handling) but not from any structural limit.

### Hard errors (client frames)

**A malformed client frame closes the connection.** Janus sends a
WebSocket Close frame whose code names the class and whose reason
text is the precise, positioned error, then closes the TCP
connection. The connection does **not** survive. Rationale: a client
speaking the protocol wrong is a broken client; executing its
well-formed frames while erroring its malformed ones is exactly the
drift-tolerance the repo rules forbid. Closing is unmissable during
development and cheap in production (the client reconnects and, if
still broken, closes again — loudly, in its own error console).

| Violation | Close code | Reason (example) |
| --- | --- | --- |
| Not valid JSON / not object-or-list-of-objects | 1008 | `frame is not a JSON object or list of objects` |
| Client-supplied `<` | 1008 | `item 0: "<" is stamped by janus; clients never send it` |
| Client-supplied `>` (reserved), `!` (exact key), `*` | 1008 | `item 0: ">" is reserved` / `"!" is janus-to-client only` / `"*" is delivery-direction only` |
| Bad sigil value shape (`@`/`+`/`-`/`<` not a string array, `?`/`*` not a string) | 1008 | `item 1: "@" must be an array of strings` |
| Present-empty `@` | 1008 | `item 0: "@" must contain at least one target` |
| Client membership `@` resolves beyond sender | 1008 | `item 0: client "+" and "-" may mutate only the sending connection` |
| Illegal channel name in `@`/`+`/`-` | 1008 | `item 0: "+" entry "lobby" is not a channel (want /-prefix)` |
| Illegal event name | 1008 | `item 0: key "9lives" is not a sigil or legal event name` |
| Duplicate JSON key | 1008 | `item 0: duplicate key "chat"` |
| Bare and suffixed form of one event anywhere in the frame | 1008 | `items 0 and 2: event "chat" appears as both "chat" and "chat!"` |
| Routing/provenance-only no-op | 1008 | `item 0: frame has no event, membership mutation, kick, or ping` |
| Structural limit exceeded (table above) | 1008 | `item 0: "@" has 65 targets (max 64)` |
| Frame over `max_frame` | 1009 | `frame exceeds 65536 bytes` |
| Binary frame | 1003 | `binary frames are not supported` |
| `+` past `max_channels` | 1008 | `join would exceed 128 channels` |

The key exactly `"!"` always carries a pong payload and is never a
zero-length-name event. Multiple suffixes such as `chat!!` fail the
event-name grammar. The WebSocket close reason is capped at 123 bytes by RFC 6455; error
messages put the position and rule first so truncation never costs
the identifying part. Nothing from a rejected frame executes
(validate-whole-then-execute, above). The close is also reported to
the tenant as a normal `close` bridge (lifecycle below) carrying the
close code, so the tenant sees its broken clients.

Silent tolerance is a bug by contract: there is no "ignore unknown
sigils" path, no best-effort delivery of the valid half of a frame,
and no coercion of near-miss value shapes.

## Identity and connection lifecycle

How a connection id is established, concretely:

1. **Upgrade request arrives** on a hub-enabled site at the hub path
   with `Upgrade: websocket`. The handler intercepts it before cache
   and upstream selection (like ping: site-scoped short-circuit).
   Host resolution runs as always: unknown host → 404, worker
   untouched.
2. **Admission checks and slot reservation, atomically:** effective hub off for the site →
   the request is not intercepted (falls through to the data plane
   like any other path). Origin check (security section) fails →
   **403**, no upgrade. The app's reserved-plus-registered connection
   count at its resolved cap → **503** +
   `Retry-After`, no upgrade. App has no registered `bridge_path` →
   **503** + `Retry-After` (hub-enabled site, hub-unready tenant —
   loud, not a hang). A successful admission reserves one slot before
   the open bridge begins, closing the concurrent-handshake race.
3. **Janus mints the connection id** (16 × `[a-z0-9]`, crypto/rand)
   and sends the **open bridge** POST (bridge contract below) while
   the handshake holds unanswered. The client's socket is not yet
   upgraded; nothing is buffered.
4. **Tenant answers 2xx** → Janus rechecks the hub tombstone while
   holding the hub lock. If DELETE or TTL reap tears the hub down
   during the bridge, Janus releases the reserved slot and rejects the
   still-unupgraded handshake with **503** + `Retry-After`. Otherwise
   Janus registers the connection, completes the upgrade (101), starts
   its writer and 30s protocol-ping clock, processes the open-response
   directives in its context, and only then starts the inbound reader
   — the tenant's chance to enroll the
   connection into channels (`{"+":["/user/42","/lobby"]}`) or greet
   it precedes all client frames. Any non-2xx answer, bridge failure,
   or client departure before 101 releases the reserved slot; no
   connection is registered.
5. **Steady state:** client text frames validate and execute at the
   edge, then forward to the text bridge. Deliveries fan in through
   the per-connection outbound queue. `?` answers at the edge.
6. **Close** — any of: client closes; protocol-ping deadline lapses;
   a hard error closes; a `*` directive closes; teardown (app
   DELETE/reap) closes. Cleanup is synchronous and local: the
   connection leaves every channel it is in (bidirectional maps,
   empty channels deleted), the writer stops, queued outbound frames
   are dropped, and the **close bridge** fires (best-effort) with the
   close code and reason. After cleanup the id is dead; deliveries
   targeting it skip and count.

**There is no user identity at this layer.** The connection id is
the only principal the hub knows. The tenant, which authenticated
the open bridge (it saw the client's cookies), maps connections to
its user model with channels — `/user/42` joined at open is the
idiomatic "all of user 42's tabs" address. This keeps authentication
entirely tenant-owned: Janus never validates a session, it validates
a socket.

A client that wants out just closes the socket; there is no leave-
everything verb (`-` with every channel works, but closing is the
normal path and cleans up identically).

## Membership model

Conceptually, per app:

| Structure | Content |
| --- | --- |
| `conns` | connection id → connection state (socket writer, outbound queue, channel set, handshake-header snapshot, bridge queue) |
| `channels` | channel name → member set (connection ids) |

The two are mirrors (bam.cr's bidirectional bookkeeping, kept):
`conns[c].channels` contains `/x` **iff** `channels["/x"]` contains
`c`. Every mutation — join, leave, disconnect — updates both sides
under the app hub's lock. A channel whose member set empties is
deleted from `channels`; there are no zombie channel entries. A
connection entry exists from the successful tombstone-check/101
transition to close cleanup.

Join/leave semantics follow the canonical execution order:

- On the **client plane**, `+` and `-` mutate only the sending
  connection. `@` remains the event-delivery set and never delegates
  membership authority. If it resolves beyond exactly the sender in
  an object that also carries `+` or `-`, validation closes the client
  with 1008 before any frame effect.
- On the **trusted planes**, `@` selects membership subjects. A direct
  id selects that live connection; a channel subject expands once to
  its post-prior-mutation member set. This cross-connection and
  channel-expansion authority belongs only to bridge responses and
  publish.
- `+` adds each subject to every named channel. Joining an existing
  membership is an idempotent no-op. `-` mirrors it; leaving a missing
  membership is an idempotent no-op. Empty channels are deleted.
- Validation computes every resulting per-connection channel count
  before execution. If any subject exceeds `max_channels`, the
  **whole frame** rejects before any mutation or delivery: client
  closes 1008, publish returns 400, and a bridge response is dropped
  whole and counted as garbage.

Membership state is **memory-only and Janus-owned**, like the
registry. The tenant holds no authoritative copy; it observes joins
it issued and frames it bridges, and it can query the snapshot
endpoint (observability below). On a Janus restart everything —
connections, channels, membership — is gone, and clients reconnect
(reload interaction below).

### Multi-tenancy: the per-app namespace

**DECIDED: channels and connection ids are implicitly namespaced per
app by Janus internally — no wire-visible change.** Concretely: hub
state is a map `app id → hub`, and every operation begins by
resolving its app — a client frame through the connection (which is
bound to exactly one app at upgrade, via host resolution), a bridge
response through the bridged connection, a publish through the
`{id}` in its URL. `/global` in tenant A and `/global` in tenant B
are entries in two different maps; no grammar, no directive, and no
key collision can make one touch the other. Connection ids likewise:
a direct-id target is looked up only in the frame's own app's
`conns`.

Within the current flat control-listener trust domain, any authenticated
control client can list app ids and address the two new Hub surfaces:
publish injects into an app and snapshot reveals channel names and
counts. These surfaces add cross-tenant reach beyond registry CRUD;
the random app-id suffix is routing, not authorization.

When host-claim scoping lands, **publish and snapshot each require an
explicit app-ownership authorization check** against the caller. They
do not inherit scoping merely because their URLs contain an app id.
Until that check exists, operators must treat every token accepted by
a control listener as trusted across all apps visible on that listener.

Host re-claim is the same story as the cache's O5: an app deleted
(or reaped) and its host re-registered under a new app id gets a
**fresh, empty hub** — teardown on DELETE/reap closes every socket
and drops every channel under the old id, and the new id's map
starts empty. A stale client reconnecting to the re-claimed host
lands in the new app's hub and is bridged to the new tenant, which
accepts or rejects it at open.

## Delivery semantics

Delivery uses stage 3 and stage 4 of the canonical execution order.
For each object, Janus takes `@` or the plane default, expands each
channel against post-mutation membership, resolves each direct id,
and unions all entries into one recipient set. Missing channels and
dead ids contribute no recipient and increment `unknown_targets`; a
connection reached through several entries receives one bundle.

Janus applies sender exclusion per event key, stamps array-valued `<`
on client-originated delivery or passes through trusted provenance,
serializes each required bundle variant once, and enqueues shared
bytes to recipient outbound queues. Validation has already rejected a
bare/suffixed collision for one event name.

Delivery to a **dead** connection cannot happen (cleanup removed it
from `conns` and every channel under the lock); a racing fan-out
that captured the member list a moment earlier finds the queue
closed and skips it — dropped, counted, never an error. Delivery to
a **slow** connection is the backpressure section.

A leave racing an unrelated fan-out linearizes at the same hub lock:
if the leave wins, the post-mutation snapshot excludes the connection;
if fan-out wins, that already-captured delivery may enqueue once.
There is no retroactive queue removal.

A delivered event bundle contains exactly: the event keys (suffix
stripped), plus `<` when present. No `@`, no `+`/`-`, no `?`, no
transport metadata. The recipient cannot see who else received it or
through which channel — a deliberate opacity; apps that need that
context put it in the payload.

### Ordering guarantees

Stated honestly:

- **Per-connection FIFO, both directions.** One sender's frames are
  read, validated, and executed sequentially by that connection's
  reader; each recipient's deliveries leave through one writer
  draining one queue in enqueue order. If connection A sends frames
  1 then 2 to the same target set, every common recipient sees 1
  before 2.
- **No cross-connection ordering.** Frames from A and B fanning to
  the same channel interleave arbitrarily per recipient; two
  recipients may see opposite orders. No sequence numbers cross the
  wire.
- **Trusted-plane directives interleave arbitrarily** with client
  fan-out. A text-bridge response's deliveries land whenever the
  tenant answers — after (usually well after) the edge delivery of
  the frame that triggered it. Within one publish request, objects
  preserve body order within each canonical stage; across publish
  requests, no ordering.
- **Membership and delivery use the canonical execution order.**
  Across frames and planes, a mutation racing a fan-out either makes the snapshot or does
  not — the member-set read under the lock is the linearization
  point.
- **No delivery receipts, no acks, no replay.** An enqueued bundle
  that dies with its connection is gone. The hub is realtime
  signaling, not a message queue (non-goals).

### Backpressure: the slow consumer

Designed, not deferred. Every connection has one outbound queue and
one writer goroutine; fan-out **enqueues and returns** — a sender is
never blocked by a recipient's socket, and one wedged recipient
cannot stall a channel.

The queue is capped two ways, whichever trips first:

| Cap | Default | Fixed in v1 |
| --- | --- | --- |
| Queued messages per connection | 256 | yes |
| Queued bytes per connection | 1 MiB | yes |

**On overflow, the connection is CLOSED, not trimmed:** WebSocket
Close code 1013, reason `slow consumer`, then cleanup (channels
left, close bridge fired with that reason). Drop-versus-disconnect
is decided for disconnect because a dropped frame is a **silent
gap**: this protocol has no sequence numbers, no replay, and no
revalidation — a client that missed a frame holds a corrupt view of
whatever state the frames were building, forever, with no signal.
Disconnection is loud, forces a reconnect, and the reconnect
re-enrolls through the open bridge, giving the tenant its hook to
resynchronize the client. Dropping would be the hub's version of the
silent miscompile. (The close itself consumes no queue space — the
Close frame is written by the closer, not enqueued.)

The full retained-data budget per connection is:

| Component | Cap |
| --- | --- |
| Outbound delivery queue | 1 MiB |
| Bridge text FIFO | 32 × `max_frame`, at most 2 MiB at the 64 KiB default |
| Frozen handshake-header snapshot | 32 KiB |

That is about **3.03 MiB per connection**, plus socket, goroutine,
map, and queue-node overhead. At the default 4096-connection app cap,
the adversarial retained-data ceiling is about **12.1 GiB**.
Shared serialized delivery buffers reduce common-case allocation
but do not weaken this sizing bound. Raising `max_conns` or
`max_frame` requires proportional memory planning.

The protocol-level ping cadence (30s) bounds how long a dead-but-
unclosed peer can accumulate queue: TCP buffers fill, the queue
fills, and the slow-consumer close fires — typically well before the
ping deadline catches it. The two mechanisms are
independent on purpose: pings catch the silent-dead, the queue caps
catch the alive-but-drowning.

## The bridge contract

Bam.cr POSTs every socket event to one hard-coded backend URL with
a `Sec-WebSocket-Frame: open|text|close` header and processes the
response body as directives. The Janus equivalent keeps that shape
and rides the existing data plane.

**Registration is hot-plane.** The app record gains one optional
field, `bridge_path`:

- `POST /1.0/apps` accepts `"bridge_path": "/rt/bridge"` alongside
  `name` and `hosts`; `PATCH /1.0/apps/{id}` can set, change, or
  clear it (`null` clears). `GET` surfaces it.
- Validation: must start with `/`, ≤ 256 bytes, no whitespace or
  control characters, no `?` or `#` (it is a path, not a URL) —
  else **400** naming the rule.
- No `bridge_path` registered → hub handshakes on that app answer
  503 (lifecycle step 2). Cold config never carries the path: which
  URL the tenant serves is tenant knowledge, exactly like socket
  paths — cold admits the capability, hot wires the tenant.

**Transport: bridge POSTs are synthetic data-plane requests.** A
bridge request enters the same machinery as a client request to the
app — upstream selection (`least_conn`, unix), passive health,
marked-503 retry (`Rip-Worker-Busy` / `Rip-Worker-Draining`), and
the doorbell: a bridge POST arriving during a dirty window **is a
ring-triggering demand**, but bridge requests use a separate,
lower-priority budget of **16 holders per app**. The ordinary client
HTTP budget remains 64 holders and bridge observation can never
consume it. Both classes share the app's ring single-flight and
three-ring bound; client admission wins when fresh capacity is
available. Bridge bodies are small and
fully buffered by Janus before sending, so every bridge POST is
replayable across marked 503s. Bridge requests are **excluded from
the micro-cache** by method (POST never caches) and never intercepted
by ping or hub (they carry no Upgrade).

Deliberately NOT reused: the tenant answering a bridge is not
"routable" in the client sense — a bridge POST when `upstreams` is
empty fails immediately (down on purpose; nothing to wake), same as
a client request's 503.

### Request shapes

All bridge requests are `POST {bridge_path}` with:

| Header | Value |
| --- | --- |
| `Sec-WebSocket-Frame` | `open` \| `text` \| `close` (the Bam header, kept) |
| `Janus-Hub-Client` | the connection id |
| `Janus-Hub-App` | the app id |
| `Content-Type` | `application/json` (text, close) / absent (open — no body) |
| plus | the connection's **handshake header snapshot** (32 KiB maximum after filtering) |

The handshake snapshot is the client's original upgrade-request
headers minus hop-by-hop and WebSocket-mechanics headers
(`Connection`, `Upgrade`, `Keep-Alive`, `TE`, `Trailer`,
`Transfer-Encoding`, `Sec-WebSocket-*`), captured once at open and
replayed on **every** bridge POST for that connection — bam.cr's
model, kept, because it is how the tenant authenticates every frame
(the `Cookie` / `Authorization` that opened the socket ride along
for its lifetime). `X-Forwarded-For` is set by the proxy layer as on
any data-plane request. A snapshot is frozen state: a client whose
session expires mid-connection keeps bridging with the old headers —
the tenant's remedy is to kick (`*`) when its own session logic says
so; Janus does not interpret cookies.

Janus measures the serialized filtered header names and values before
sending the open bridge. A snapshot over 32 KiB rejects the handshake
with **431 Request Header Fields Too Large**; it is never truncated.

Per frame type:

- **open** — body: empty. Sent before the 101; the tenant's answer
  is the admission decision.
- **text** — body: the client's frame **verbatim** — the exact bytes
  received on the socket, already validated by the edge, never
  re-serialized (no silent normalization; the tenant sees what the
  client said, including its `!` spellings and its `@`).
- **close** — body: `{"code": 1001, "reason": "…"}` — the WebSocket
  close code and reason as observed or generated by Janus (client
  close, ping timeout, slow-consumer close, hard-error close, `*`
  kick). Sent after local cleanup completes.

**Per-connection bridge FIFO.** Each connection's bridge POSTs are
serialized: open, then texts in read order, then close. The
serialization queue is bounded (32 pending and 32 × `max_frame`
bytes); overflow drops the
**oldest queued text** and counts it (`bridge_dropped`) — never
blocks the reader, never closes the socket (failure modes below for
why text bridging is at-most-once).

Local close stops new text admission, discards and counts queued
texts, waits at most for the one in-flight POST, and then gives the
close notification one bounded attempt. The whole post-close drainer
has a **40-second deadline** (one 20-second in-flight bound plus one
20-second close bound). `close` is lifecycle-critical: it is never
silently lost behind text or reordered before an in-flight text.
Expiry or failure increments `bridge_failed`, logs the undelivered
close with app and connection attribution, and terminates the
drainer; guaranteed close delivery remains a non-goal. Across connections,
bridges run concurrently, bounded only by the data plane's normal
capacity behavior (a slow tenant sheds via marked 503s and the
bounded queue, not via unbounded goroutine pileup).

### Response processing

The tenant's answer to any bridge POST:

| Tenant answer | Janus does |
| --- | --- |
| 2xx, empty body (or 204) | Nothing — the frame was observed |
| 2xx, JSON body | Execute the body as directive objects on the **bridge-response plane**, in the bridged connection's context |
| Anything else | The failure table below |

Bridge-response directives get the trusted-plane policy from the
sigil table: `<` and `*` legal, `?`/`!`/`>` rejected, `@` defaults
to the originating connection. For an **open** response, "the
originating connection" is the connection being admitted — so a
body of `{"+": ["/user/42", "/lobby"]}` enrolls it, and a body of
`{"@": ["/lobby"], "joined": {"who": "…"}}` announces it. For a
**close** response, the originating connection is already gone:
directives still execute (the tenant can announce the departure to
channels), and any delivery that resolves to the dead connection
skips and counts, per the delivery rules.

A malformed bridge-response body — non-JSON, illegal grammar,
plane-violating sigils, structural limits — is **tenant garbage**:
the directives are dropped **whole** (validate-then-execute, nothing
partial), a counter increments (`bridge_garbage`), and the error is
logged with app id, connection id, frame type, and the positioned
parse error. The client is unaffected — its frame already executed
at the edge. Garbage is never repaired and never partially applied;
bam.cr's bare `rescue puts "invalid response from backend"` becomes
a counted, attributed, structured event.

The 2xx status is otherwise uninterpreted: 200 and 201 and 299 are
the same answer. Response headers are ignored. The response body cap
is the data plane's normal response handling; directive bodies are
read fully before execution with a 1 MiB read cap — a response body
larger than that is tenant garbage (counted, logged), not a
streaming case.

### Failure modes

"Failure" = the data-plane attempt did not produce a 2xx: tenant
5xx, non-2xx, transport error, dial failure, all-upstreams-
unhealthy, empty upstreams, doorbell hold expiry, or the bridge
timeout (one bound per bridge POST: **20s**, sized to contain one
full ring hold (17s) plus margin; a bridge held across additional
rings times out at 20s and fails per this table).

The policy differs by frame type because the frame types have
different jobs:

| Frame | Role | On failure — exact client-visible behavior |
| --- | --- | --- |
| **open** | Admission gate (synchronous) | No connection is registered. Tenant answered a non-2xx status → Janus forwards **that status and body** to the client as the plain-HTTP answer to the upgrade request (a 401/403 from the tenant reaches the client honestly; marker headers stripped as on the data plane). Transport failure / timeout / empty upstreams / unhealthy → **503** + `Retry-After`, no upgrade. Every path releases the reserved slot; a late 2xx after teardown fails the tombstone recheck the same way |
| **text** | Observation + reaction hook (asynchronous) | **Nothing client-visible.** The frame already executed at the edge; what failed is the tenant's view of it. The failure is counted (`bridge_failed`), logged with attribution, and the POST is **not retried** (beyond the data plane's own marked-503 retry within the attempt) — the text bridge is **at-most-once observation**, stated as contract, not discovered in an outage |
| **close** | Notification (best-effort) | Nothing client-visible (the socket is already closed). Counted and logged; no retry queue in v1. Consequence stated honestly: a tenant that keeps per-connection state learns of the close late — at its next failed reaction to that connection, from the membership snapshot endpoint, or by its own liveness logic. Guaranteed close delivery is a non-goal |

Why text-bridge failure must not close the socket: during a reload
dirty-window or a tenant crash, closing every sender would convert a
worker-plane outage into a client-plane massacre — precisely
inverted from the edge-terminates-sockets promise. The hub keeps
fanning out through tenant outages; the tenant's view is lossy under
its **own** outage, which is the only honest option that keeps the
promise. Apps that need the server to reliably see a message send it
as a normal HTTP request (the entire rest of Janus) and use the hub
for the fan-out.

The tenant-down window in numbers: fan-out unaffected; joins/leaves
unaffected; `?`/`!` unaffected; new connections rejected 503 at
open (dirty window: held up to the 20s bridge timeout first, so a
reload usually admits with a one-boot delay rather than rejecting);
text observation dropped and counted. Recovery is automatic: the
next bridge POST that finds a published pool succeeds; nothing needs
re-arming.

## The publish plane

`POST /1.0/apps/{id}/hub/publish` on the existing control listeners
— the hot-plane injection point (`bampub.cr`'s `/publish`,
namespaced per app and moved behind the control plane).

- **Auth posture: exactly `/1.0`.** The endpoint exists on every
  configured control listener (internal / local / public) and
  inherits that listener's Bearer-token behavior. No additional
  per-call auth in v1; per-app publish secrets (a public publish
  surface) are a non-goal until a use case demands them — the
  Phase 7 spec line's "optional public publish with secret later"
  stays later. Within that current flat trust domain, publish is a
  cross-app capability. Host-claim scoping requires an explicit
  caller-owns-app check here; app-id routing alone does not provide it.
- **Body:** one directive object or a list, same grammar, publish-
  plane policy (sigil table): `@` **required** per object (there is
  no originating connection to default to — an object whose
  directives need targets and carries no `@` is a 400, where bam.cr
  treats as a silent no-op). A present-empty `@` is also **400**.
  Array-valued `<` and `*` are legal; `?`, `!`, `>` are 400.
  Structural limits as in the limits table; body cap is the control
  plane's existing 1 MiB.
- **Execution:** use the canonical execution order on the app's hub.
  Sender exclusion never applies (no
  originating connection — `!`-suffix rule 3), so bare and `!`
  event spellings deliver identically here.
- **Responses:** `200` with `{"objects": N, "deliveries": M,
  "unknown_targets": K}` (deliveries = bundles enqueued; a
  slow-consumer close racing the enqueue can still lose one — the
  counts are enqueue-time truth, not receipts). Unknown `{id}` →
  **404**. Malformed body → **400** with the positioned error
  (object index, key, rule). App exists but hub never enabled on any
  of its sites → **409** `hub is not enabled for any site of this
  app` — publishing into a hub that cannot have members is a caller
  bug, said loudly.
- The publish handler touches only hub state — never `upstreams[]`,
  never the doorbell. A publish while the app is mid-reload delivers
  normally: membership state rides above the worker plane.

Registration of `bridge_path` and the publish endpoint are both hot;
nothing about the hub's tenant wiring lives in cold config. Cold
declares *that* a site speaks hub; hot declares *who* is listening.

## Cold config

Site-scoped capability, the **ping**/**cache** pattern: normal
Caddyfile grammar, a directive with optional `on|off` and an
optional block.

```caddyfile
{
	janus {
		control local
		hub {                   # global override: on, tuned
			path /realtime      # WS endpoint path (default /hub)
			max_conns 4096      # connections per app
			max_frame 64kb      # largest client frame
			max_channels 128    # channels per connection
			origin same         # Origin policy: same | any | host…
		}
	}
}

(ripdev_tls) {
	tls certs/ripdev.io.crt certs/ripdev.io.key
}

*.ripdev.io {                   # catchall: inherits hub on
	import ripdev_tls
	janus
}

api.ripdev.io {
	import ripdev_tls
	janus {
		hub off                 # explicit off beats inherited on
	}
}

chat.ripdev.io {
	import ripdev_tls
	janus {
		hub {
			max_conns 16384     # override one key; the rest inherit
			origin same chat-admin.ripdev.io
		}
	}
}
```

### Legal lines

```text
hub
hub on
hub off
hub {
	path <path>              # starts with /; no ? or #
	max_conns <n>            # integer ≥ 1
	max_frame <size>         # ≥ 1kb
	max_channels <n>         # integer ≥ 1
	origin same | any | same <host>… | <host>…
}
hub on { … }                 # same as hub { … }
```

Interception rule: on a hub-enabled site, a request to `path`
carrying `Upgrade: websocket` is taken by the hub (before cache and
upstream selection, after ping). A **non-upgrade** request to the
same path is not special — it flows through the data plane like any
path; the hub claims upgrades only. Janus resolves the per-app
`max_conns` cap as the **minimum effective value across every
hub-enabled host claimed by that app**. Every admission checks that
one floor against reserved-plus-registered slots, regardless of the
arriving host. Adding a tightly capped host can lower the app's floor;
connections above a newly lowered floor remain alive, while new
admissions reject 503 until usage falls below it. Operators size the
full per-connection memory budget against this app-level floor.

### Cascade

Yes — global default → site override, the standard rules:

| Global | Site | Effective |
| --- | --- | --- |
| (unset) | (unset) | **off** |
| `hub` / `hub on` | (unset) | **on** (global tuning) |
| `hub` | `hub off` | **off** |
| `hub off` / (unset) | `hub on` | **on** |
| `hub { max_conns 4096 }` | `hub { max_conns 16384 }` | site value 16384; an app spanning both effective values is capped at 4096 |

Tuning subdirectives cascade per key: a site block overrides only
the keys it names; unmentioned keys inherit the global values;
built-in defaults apply when unset at every level. There are no
process-wide-only keys (contrast cache's `max_bytes`): every hub
knob is per-site admission policy.

### Defaults

| Knob | Default |
| --- | --- |
| capability | off (unset at every level) |
| `path` | `/hub` |
| `max_conns` | 4096 (per app) |
| `max_frame` | 64kb |
| `max_channels` | 128 (per connection) |
| `origin` | `same` |
| Outbound queue caps | 256 messages / 1 MiB per connection (fixed in v1) |
| Bridge queue | 32 pending and 32 × `max_frame` bytes per connection |
| Bridge timeout | 20s per POST (fixed in v1) |
| Post-close bridge drainer | 40s total (fixed in v1) |
| Filtered handshake-header snapshot | 32 KiB per connection (fixed in v1) |
| Bridge doorbell holders | 16 per app, separate from 64 client holders (fixed in v1) |
| Protocol ping cadence | 30s (fixed in v1) |
| Structural limits (targets, objects, keys, name lengths) | Limits table (fixed in v1) |

### Hard errors (reject at parse)

- Unknown argument (anything but `on` / `off`)
- `hub off { … }` — a block on an off switch is a contradiction
- Unknown subdirective inside the block
- Duplicate `hub` directive in the same block; duplicate subdirective
- `path` missing an argument, not starting with `/`, or containing
  `?`, `#`, whitespace, or control characters
- `max_conns`, `max_channels` not a positive integer
- `max_frame` not a valid size or under 1kb
- `origin` with no arguments; `origin any` combined with anything
  else (`any` is total); an `origin` host that is not a plausible
  hostname (registry host validation reused)
- Nested block under any subdirective

## Reload interaction

The pool protocol's events, one by one, with the hub's exact
behavior at each:

| Event | Open sockets | Membership | In-flight bridges | New handshakes |
| --- | --- | --- | --- | --- |
| **Doorbell PUT (admission cut)** | Untouched | Untouched | Already-sent POSTs complete against old workers (finishing a commitment); newly-queued POSTs select the doorbell and ring/hold | Open bridge rings the doorbell and holds (ring caps), then admits against the fresh pool |
| **Dirty window (boot in progress)** | Untouched; fan-out, joins, `?` all normal | Untouched | Text bridges hold up to the 20s bridge timeout; queue overflow drops oldest text (counted); close bridges hold the same way | Held then admitted, or 503 past the caps |
| **Boot failure** | Untouched | Untouched | Text/close bridges fail per the failure table (counted, no client impact) | Open bridge receives the 503 + boot error → handshake rejected with it |
| **Sockets PUT (pool published)** | Untouched | Untouched | Held bridges complete against fresh workers | Normal admission |
| **`PUT upstreams []` (down on purpose)** | Untouched | Untouched | Fail fast per failure table | 503 immediately |
| **PATCH hosts** | Connections on retained hosts stay open; connections whose bound host is removed close through the internal close-with-reason path, Close 1001 `host removed` | Removed-host connections leave all channels; all other membership stays | Removed-host close notifications get the bounded best-effort attempt | Added hosts admit under their effective policy; removed hosts stop resolving to the app |
| **DELETE / heartbeat reap** | **All closed through the internal mechanism:** final reason `app deregistered`, then Close 1001 ("going away") | Dropped entirely | Close notifications get their bounded best-effort attempt | 404 (host unknown once freed) |

Two load-bearing consequences, stated plainly:

- **An app reload is invisible to connected hub clients.** Sockets,
  channels, and fan-out ride above the worker plane; the only
  reload-visible artifact is delayed (or, past the ring caps,
  dropped) tenant *observation* of text frames, and a one-boot delay
  on new handshakes. Heartbeat ≠ readiness keeps the registration —
  and therefore the hub — alive through every dirty window.
- **Hub teardown keys off registration lifecycle, never off
  `upstreams[]`.** Upstreams PUTs (doorbell, sockets, empty) do not
  touch hub state — no analog of the cache's purge-on-PUT exists
  here, deliberately: membership is client state, not
  generation-scoped response bytes. Only DELETE and TTL reap — the
  events that kill the registration itself — tear the hub down.

**Janus restart** is the registry story, extended: apps vanish,
sockets die (the process died), membership dies with them. Recovery
order: the tenant re-registers and re-publishes upstreams (existing
contract), then clients reconnect — each reconnect is a full open
bridge, so the tenant re-authenticates and re-enrolls every
returning connection; membership rebuilds from truth, not from a
snapshot. Reconnect stampede is bounded by `max_conns`, by the
tenant's own capacity (open bridges shed via marked 503s → 503 at
the handshake), and by client backoff (client responsibility, stated
in tenant docs). Nothing hub-side persists, matching the registry's
memory-only contract — durability is a non-goal.

**Caddy config reload** (cold): `App.Provision` acquires a pooled
process-state holder from `caddy.UsagePool` or constructs it once. The
holder contains the live registry/host index and its hub set; hub
entries are keyed by stable app identity. Newly provisioned handlers
therefore bind to the same registry and hub entries before the old
config is retired, instead of constructing a split-brain registry.
The pooled holder owns the registry sweeper. Releasing an old module
reference neither stops that sweeper nor tears down a hub; final
process cleanup releases the holder.
Only registry DELETE or TTL reap deletes its pooled app entry.
Upstreams PUT and Caddy reload never do.

A live socket therefore remains upgraded, keeps its connection id,
channel memberships, outbound FIFO, bridge FIFO, and fan-out ability
across a successful Caddy reload. New admissions use the new cold
policy and the app's recomputed minimum `max_conns` floor. A host whose
effective Hub capability flips to **off** stops intercepting upgrades
and closes sockets bound through that host via Janus's internal
close-with-reason mechanism, then Close 1001 `hub disabled`. This
internal mechanism is always present and is separate from the
wire-visible trusted-plane `*` tenant kick.

## Security

**Origin policy (cold, default `same`).** Browsers attach `Origin`
to every WebSocket handshake and scripts cannot forge it —
cross-site WebSocket hijacking is real and the default must close
it. `origin same`: the `Origin` header must be present and its host
must equal the request's Host (scheme is not compared; the site is
TLS-terminated already). Extra hosts may be allowlisted
(`origin same admin.example.com`). `origin any` disables the check —
for token-authenticated non-browser APIs — and is a deliberate,
spelled-out opt-out. Failure → **403** before any bridge is sent.
Non-browser clients that send no `Origin` header fail `same` — an
allowlisted-host or `any` posture is the knob for them, chosen by
the operator, never silently defaulted.

**Identity and authentication.** A connection's identity is its
minted id (~82 bits, crypto/rand); authentication is the tenant's
open-bridge decision on the handshake headers. Janus never inspects
sessions. The id doubles as an address; possession of an id permits
direct-targeting it — acceptable because ids are unguessable,
disclosed only to the owning client (its own deliveries carry them
in array-valued `<`), the tenant through bridge headers, and whoever
either intentionally tells. Membership snapshots never expose raw ids.

**The `<` anti-spoof rule (DECIDED).** Client-supplied `<` rejects
loudly (hard-error table); attribution on client-originated
deliveries is stamped by the edge from the socket's own identity. No
client can speak as another. Trusted planes may attribute freely —
the tenant can impersonate, which is the tenant's right in its own
app.

**Authorization capability.** An untrusted client controls only its
own membership: `+`/`-` join or leave the sending connection, while
`@` controls event delivery and never membership subjects. Any client
attempt to combine membership mutation with an `@` resolving beyond
itself closes 1008 before effects. Cross-connection mutation,
including channel expansion as a `+` subject, belongs only to the
trusted bridge-response and publish planes.

This sender-only rule prevents one client from enrolling or removing
another client. It does not make channel names an ACL: a client may
self-join any legal channel name it knows and may deliver to any
known channel or connection id. Tenants use unguessable room names
where discovery itself matters and use trusted `-` or `*` to remove
or kick clients. A per-connection grant/capability model that
authorizes specific self-joins is an explicit future non-goal.

**Resource limits, rejecting loudly at each:** connection cap per
app (503 at handshake), frame-size cap (close 1009), channels per
connection (close 1008), structural caps per frame (close 1008),
outbound queue caps (close 1013), bridge queue cap (drop oldest,
counted). Per-IP connection caps are deliberately absent in v1 (the
edge has no per-IP state anywhere else either); the per-app cap
bounds the damage and the open bridge gives the tenant a per-client
veto at admission.

**Transport:** hub sockets exist only on TLS-terminated sites (the
only kind Janus serves); control-plane publish rides the control
listeners' existing posture; bridge POSTs ride unix sockets to the
tenant. `permessage-deflate` is not negotiated in v1 (non-goals) —
no compression-oracle surface.

  Counters are the acceptance-test instrument (a delivery must not
  touch a worker — provable as `deliveries` up, worker request count
  flat modulo the text bridge). Same discipline as `/1.0/cache`:
  atomics, snapshot never blocks the hot path, monotonic but not
  mutually atomic.

- **Membership snapshot:** `GET /1.0/apps/{id}/hub` → connection
  count, channel names with member counts, and optional
  per-connection channel lists keyed by opaque snapshot handles.
  Handles are not connection ids, cannot be used as wire targets, and
  reveal no raw address. This is the tenant's resync instrument and
  `test.sh`'s membership oracle. Output contains names, counts, and
  opaque handles only, never raw connection ids or queued payloads.
  Unknown id → 404. Current control-listener authentication applies;
  host-claim scoping adds an explicit caller-owns-app check to this
  endpoint.

- **Logs:** every loud rejection (hard errors, origin failures,
  bridge garbage) logs once with app id, connection id, and the
  positioned reason — the server-side twin of the client's close
  reason. No per-delivery logging on the hot path.

## Close mechanisms

Two distinct close paths are DECIDED:

- **Internal close-with-reason is always present.** Janus uses it for
  registry DELETE, TTL reap, removed hosts, Hub disable, protocol
  errors, ping timeout, and slow consumers. It does not enter the wire
  executor and does not depend on tenant authority. It sends the
  applicable WebSocket Close reason when the socket remains writable,
  then the specified Close code, and performs local cleanup. It does
  not emit a wire `*` application frame.
- **Wire-visible `*` is a trusted-plane tenant kick.** A bridge
  response or publish object may carry a JSON string reason up to 128
  bytes. After ordinary events in that object deliver, each resolved
  target receives `{"*":"<reason>"}` as its final application frame
  and closes with code 1000. Client-supplied `*` closes 1008 as a hard
  protocol error.

Per-IP connection caps remain outside v1. `max_conns` bounds per-app
damage, the open bridge gives the tenant a per-client veto, and NAT
makes an IP cap a weak identity boundary.

## Non-goals

Enumerated; each is a decision, not an omission:

- **No subtree fan-out** — targeting `/friends` does not reach
  `/friends/high-school`; hierarchy is naming-only (DECIDED). A
  future subtree-delivery decision is explicitly deferred, default
  off.
- **No wildcards** in targets or channel names (`/game/*` is an
  illegal channel name by grammar).
- **No `>`** — reserved, rejected, unused in v1 (DECIDED). `@`
  answers "to whom"; a second targeting spelling would violate
  one-way-to-do-each-thing.
- **No binary WebSocket frames** — bam.cr never supported them;
  rejected with close 1003, not ignored.
- **No persistence, no replay, no delivery receipts** — memory-only,
  like the registry; a missed frame is missed; reconnection
  re-enrolls through the open bridge.
- **No cross-node clustering** — one process, one hub set; matching
  the registry and cache.
- **No per-connection membership grants in v1** — client self-join
  applies immediately, but only to the sender. A future capability
  model may let the tenant grant specific channel joins; the bridge
  is reactive rather than an approval gate in v1.
- **No per-app publish secrets / public publish surface** — publish
  auth is the control plane's, "later" per the Phase 7 spec line.
- **No `permessage-deflate`** — no compression negotiated in v1.
- **No user identity at the edge** — connections, not users;
  `/user/N` channels are the tenant's mapping.
- **No client-visible channel enumeration** — no list-members or
  list-channels verb on the client plane; the snapshot endpoint is
  control-plane only.
- **No rate limiting beyond the caps stated** — no token buckets,
  no per-IP counters in v1.
- **No message queue semantics** — the text bridge is at-most-once
  observation; commands that must reach the server reliably are
  HTTP requests.

## Acceptance sketch (`test.sh` hub group)

The instrument: a fixture tenant with a bridge endpoint that records
every POST it receives (headers, frame type, body) and answers from
a scriptable playbook (2xx + directives, non-2xx, garbage, delay),
plus `/1.0/hub` counters and the membership snapshot. A small WS
client drives sockets. Group lands after `tls`, before `tenant`, in
`test.sh`'s group order.

| Case | Assert |
| --- | --- |
| **Open handshake, full path** | Fixture receives `open` with `Sec-WebSocket-Frame: open`, `Janus-Hub-Client`, `Janus-Hub-App`, cookies from the handshake; answers `{"+":["/room"]}`; 101 completes; snapshot shows the connection in `/room` |
| **Open rejected by tenant** | Fixture answers 403 + body; client receives 403 + that body, no upgrade; `conns` unchanged |
| **Open with tenant down** | Empty upstreams → handshake 503 + `Retry-After`; no bridge retry |
| **No bridge_path** | Hub-enabled site, registered app without `bridge_path` → handshake 503 |
| **Bare-name exclusion (`!` rule 1)** | Two connections of one "user", A1 sends `{"@":["/room"],"chat":{}}` with A1,A2,B in `/room`; A2 and B receive, A1 does not |
| **`!` includes sender (`!` rule 2)** | A1 sends `chat!`; A1, A2, B all receive; every recipient sees key `chat`, never `chat!` |
| **Publish ignores exclusion (`!` rule 3)** | Publish `{"@":["/room"],"chat":{}}` and `chat!` variant → identical delivery sets |
| **`?` is pong, `chat!` is not (`!` rule 4)** | `{"?":"t1"}` → `{"!":"t1"}` from the edge; fixture receives the text bridge but no worker answers the pong (worker counter flat) |
| **`<` stamped** | B receives A's frame with `"<":[A-connection-id]`; a trusted relay may append provenance |
| **`<` spoof rejected** | A sends `{"<":["x"],"chat":{}}` → close 1008 with positioned reason; nothing delivered; close bridge carries 1008 |
| **`>` reserved / exact `!` / client `*` rejected** | Each → close 1008, precise reason |
| **Join/leave bookkeeping** | `+` then snapshot shows both directions; `-` empties → channel gone from snapshot |
| **Default target is sender** | `{"+":["/x"]}` with no `@` joins the sender; snapshot confirms |
| **Client membership is sender-only** | A sends `{"@":["B"],"+":["/x"]}` → A closes 1008, neither A nor B joins, and no event delivers; trusted publish can enroll B |
| **Channel vs id discrimination** | Target `/room` fans to members; target `<conn id>` delivers directly; target `room` (no slash) in `+` → close 1008 |
| **Naming-only hierarchy** | Members in `/a/b`; publish to `/a` → zero deliveries, `unknown_targets` counted |
| **Per-app isolation** | Two apps, same channel name; publish into A → B's member receives nothing |
| **Text bridge observation** | A sends 3 frames; fixture receives 3 texts, in order, bodies byte-identical to the wire frames |
| **Bridge response directives** | Fixture answers a text with `{"@":["/room"],"note":{}}` → delivered to `/room` in the sender's context (bare name excludes the sender) |
| **Bridge garbage** | Fixture answers 200 + non-JSON → `bridge_garbage` counted, sender unaffected, nothing delivered |
| **Atomic grammar rejection** | Duplicate keys, `chat` + `chat!`, `chat!!`, empty `@`, routing-only no-op, or `max_channels` overflow rejects the whole frame before membership or delivery |
| **Text-bridge failure is invisible** | Fixture 500s texts → deliveries still fan out at the edge; `bridge_failed` counted; sockets stay open |
| **Reload invisibility** | Connect, join; doorbell PUT + sockets PUT (fixture reload); socket stays open, membership intact, fan-out works mid-window; held text bridge completes after publish |
| **Teardown on DELETE** | DELETE app → all its sockets close through the internal mechanism with Close 1001 `app deregistered`; snapshot 404s |
| **Slow consumer** | Recipient stops reading; sender floods until queue caps trip → recipient closed 1013 "slow consumer"; `slow_closes` counted; sender unaffected |
| **Oversize frame** | > `max_frame` → close 1009 |
| **`max_conns` reservation** | Cap N; N concurrent slow open bridges reserve N slots; N+1th handshake → 503; failures release slots |
| **Open/teardown race** | DELETE/reap while open bridge is pending → returning 2xx sees tombstone, receives no 101, releases slot, and registers no zombie |
| **Per-app cap floor** | One app spans hosts capped 10 and 20 → app admits at most 10 reserved-plus-registered connections from either host |
| **Origin policy** | Wrong-Origin browser-shaped handshake → 403 (no bridge sent); `origin any` site admits it |
| **Publish plane** | Publish with absent or empty `@` → 400 positioned; unknown app id → 404; hub-less app → 409; happy path → 200 with counts |
| **Publish `*` kick** | Publish `{"@":["<id>"],"*":"kicked"}` → target receives the `*` frame, then close 1000; close bridge fires |
| **Caddy reload persistence** | Reload cold config while connected → socket id, memberships, queues, and fan-out survive through the pooled registry/hub state |
| **Bridge bounds** | >32 KiB filtered headers → 431; bridge holders never consume 64 client holders; local close discards queued texts and terminates its close drainer within 40s |
| **Snapshot opacity** | Snapshot exposes counts, channel names, and non-targetable opaque handles, never raw connection ids |
| **Parse rejections** | Each cold-config hard-error line fails `caddy adapt` with a precise error |

### Go-test surface

`go test ./...` units, one per contract row:

- **Frame validation table** — one test per hard-error row (each
  malformed input → the exact close code and a reason matching the
  positioned form), plus accepts for every legal example in this doc.
- **Grammar edges** — sigil-vs-event-name boundary (`!` exact vs
  `chat!` vs `!chat` illegal), name regexes (channels, events,
  length caps), structural limits at N and N+1.
- **Membership bookkeeping** — bidirectional mirror invariant after
  every op sequence; empty-channel deletion; join/leave idempotence;
  `max_channels` rejection applies nothing.
- **Target resolution and authorization** — dedupe across channel+id
  overlap; unknown target skip/count; sender-only client membership;
  trusted-only channel expansion in `+`.
- **`!`-suffix pins ×4** — originating-connection-only exclusion
  (two conns, one "user"); suffix stripped on delivery; publish
  plane ignores spelling; exclusion never keys off `<`.
- **`<` stamping** — one-element array on client deliveries,
  array-valued provenance passed through on trusted planes, relay
  append behavior, absent stays absent.
- **Executor plane policy** — every sigil × plane cell of the table
  (client `<`/`>`/`!`/`*` reject; bridge `?` rejects; publish `@`
  required; …).
- **Lifecycle** — cleanup removes both map directions, closes the
  queue, fires close bridge once; racing fan-out vs close never
  panics and never delivers post-cleanup; slot reservation, release,
  tombstone recheck, and reader-after-open ordering (`-race` on the
  package).
- **Backpressure state machine** — message cap, byte cap, close
  1013, closer-not-enqueued property.
- **Bridge client** — request shapes per frame type (headers,
  snapshot scrubbing of hop-by-hop and `Sec-WebSocket-*`), verbatim
  text body, 32 KiB snapshot rejection, per-connection FIFO,
  drop-oldest overflow, bounded close priority/drainer, separate
  doorbell holder budget, response processing (2xx-empty / 2xx-directives /
  garbage / non-2xx per type).
- **Registry extension** — `bridge_path` validation rows,
  PATCH set/clear, surfaced in GET.
- **Cold config** — parse of every legal line; every parse
  hard-error; cascade table rows (the ping/cache test pattern).
- **Teardown** — DELETE/reap closes all conns with the right
  code/reason and leaves no map entries (leak-checked by count).
- **UsagePool reload** — a newly provisioned App reuses the registry
  and app-keyed hubs; module cleanup alone never closes sockets.

### Bench plan

Per the measurement discipline (interleaved A/B, ratios over
absolutes, canonical rig — `bench/` grows a hub arm):

1. **Fan-out throughput** — one publisher, one channel, N=100/1k/4k
   subscribers on loopback TLS: sustained deliveries/s and per-
   delivery CPU; the number that sizes "how big a room can one Janus
   push."
2. **Delivery latency** — publish-to-client p50/p99 at 10%/50%/90%
   of the fan-out ceiling; assert the queue design keeps p99 flat
   until saturation, not degrading linearly.
3. **Connection ceiling and idle cost** — ramp connections to
   `max_conns` at zero traffic: RSS per idle connection (goroutine
   pair + queues + snapshot), handshake+open-bridge rate (conns/s
   admitted against a real tenant).
4. **Slow-consumer isolation** — one wedged subscriber in a 1k-member
   channel under load: other members' p99 must not move; the wedged
   one closes 1013 within the cap arithmetic.
5. **Reload under fan-out** — sustained publish traffic through a
   doorbell cut + republish: zero socket drops, zero delivery stalls
   beyond lock-hold noise, `bridge_dropped`/`bridge_failed` counted
   as designed.
6. **Text-bridge tax** — client-send throughput with the fixture
   tenant answering 204 instantly vs 5ms: quantifies the observation
   plane's cost and validates that edge delivery does not degrade
   with a slow tenant.

Measured numbers belong in the performance doc's Measured results
with their implementation commit, per rule 8.

## Registry and protocol deltas

The implementation applies these authoritative protocol deltas:

1. **The registry record** gains `bridge_path` (optional; validation
   above), settable at `POST /1.0/apps` and `PATCH /1.0/apps/{id}`,
   surfaced by GET.
2. **The control surface** gains two routes: `POST
   /1.0/apps/{id}/hub/publish` and `GET /1.0/apps/{id}/hub`, plus
   process-wide `GET /1.0/hub` — all with the existing Bearer
   behavior.
3. **The pool protocol** gains one sentence under the registry
   section: DELETE and TTL reap additionally tear down the app's hub
   (closing its WebSocket connections); upstreams PUTs never touch
   hub state.
4. **The data plane decision table** is unchanged: hub upgrade
   interception happens in the site handler before the table runs
   (like ping), and bridge POSTs enter the table as ordinary
   requests.
5. **`test.sh`** gains the `hub` group (after `tls`, before
   `tenant`); AGENTS.md's group list gains `hub` in that position;
   the build SPEC's Phase 7 acceptance box ticks against this doc.

Integration seams, named for the implementer: upgrade interception
in `Handler.ServeHTTP` (`handler.go`) alongside the ping check;
`hub` cascade resolution mirroring `provisionCache`
(`cache_config.go` pattern); `App.Provision` acquiring the shared
registry and app-keyed hubs through `caddy.UsagePool`; teardown hook invoked from
`appRegistry.delete` and `sweepExpired` (`apps.go`) exactly where
the cache purge hook is; bridge POSTs built on `dataPlane
.serveResolved`'s machinery (`dataplane.go`, `ring.go`) with a
synthetic request and a captured response; publish and snapshot
handlers registered in `controlMux` (`control_api.go`).
